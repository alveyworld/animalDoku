## 1. Problem Statement

Unit tests prove the engine is correct, but nothing verifies that the assembled app — board rendering, tap handling, view model wiring, and win overlay — actually lets a real user complete a puzzle. A regression in tap routing, cell hit-testing, or win detection could break the core loop while every unit test stays green. An automated XCUITest that plays a known puzzle from launch to the win screen guarantees the full play path works before every release.

## 2. Goals

- Add `UITests/GameplayUITests.swift` that launches the app, places the known solution for `puzzle-001`, and asserts the win screen appears.
- Drive the app through public, accessibility-identified controls only — no private hooks into game state.
- Cover the primary happy path plus one critical interaction (undo, or invalid-placement highlight) so the test guards more than just win detection.
- Make the test deterministic by launching into the known bundled puzzle via a launch argument.
- Establish accessibility identifiers on cells and controls so UI tests are stable across layout changes.

## 3. Non-Goals

- Exhaustive UI coverage of every screen or setting (targeted, not comprehensive).
- Snapshot / visual regression testing (optional, future).
- Accessibility audit (P6.3) or device matrix (P6.4).
- Testing theme switching, sound, or settings persistence (unit/manual elsewhere).
- Performance benchmarking.

## 4. Personas & User Stories

- **As a developer**, I want an automated play-to-win test so that a broken core loop fails CI before release.
- **As a release manager**, I want a green end-to-end check so that I can ship v1.0 with confidence.
- **As a player** (indirect), I want the tap-to-place-to-win flow to always work on a fresh install.

## 5. Functional Requirements

- **FR-1.** `UITests/GameplayUITests.swift` MUST exist in the UI test target (P1.3) and launch the app fresh.
- **FR-2.** The app MUST expose stable accessibility identifiers for each board cell (e.g. `cell-<row>-<col>`), the win overlay, and the Play Again / undo controls.
- **FR-3.** The test MUST launch directly into a **known** puzzle (`puzzle-001`) via a launch argument (e.g. `-uiTestPuzzle puzzle-001`) so the solution is deterministic.
- **FR-4.** The test MUST place animals at each `solution` position by tapping the corresponding cell identifiers in Place mode.
- **FR-5.** After placing the full solution, the test MUST assert the win overlay (title "Puzzle Complete!") is visible.
- **FR-6.** The test MUST assert the board is locked after win (further cell taps do not change state / no crash) per Formal Rules interaction model.
- **FR-7.** The test MUST include at least one secondary assertion: tap an occupied cell then Undo and verify the animal is removed, OR place an off-solution animal and verify a violation is surfaced.
- **FR-8.** The test MUST tap Play Again and assert the board resets to empty and the win overlay dismisses.
- **FR-9.** The test MUST NOT read or mutate `GameSession` directly; it interacts only through the UI accessibility layer.

## 6. Non-Functional Requirements

- **Performance** — Full test completes in <60s on CI simulator; no arbitrary `sleep` — use XCUIElement existence expectations.
- **Security** — N/A.
- **Privacy** — N/A.
- **Accessibility** — This test depends on accessibility identifiers/labels, reinforcing the a11y work audited in P6.3.
- **Maintainability** — Cell taps derived from the loaded puzzle's `solution` (read from a test fixture copy), not hard-coded coordinates duplicated in the test.
- **Reliability** — Deterministic; no flakiness from animations (disable or wait via expectations; respect Reduce Motion launch arg if helpful).
- **Determinism** — Fixed puzzle via launch argument; no reliance on default-puzzle changes.

## 7. Acceptance Criteria

- **AC-1.** *Given* a fresh launch into `puzzle-001`, *when* the test taps every `solution` cell in Place mode, *then* the win overlay with "Puzzle Complete!" becomes visible.
- **AC-2.** *Given* the win overlay is shown, *when* the test taps a board cell, *then* the cell state does not change (board locked) and no crash occurs.
- **AC-3.** *Given* a placed animal, *when* the test taps Undo, *then* the most recent animal is removed from the board.
- **AC-4.** *Given* the win overlay, *when* the test taps Play Again, *then* the overlay dismisses and all cells are empty.
- **AC-5.** *Given* `xcodebuild test` on the UI test target, *when* run on the CI simulator, *then* `GameplayUITests` passes without manual intervention.
- **AC-6.** *Given* the app under test, *when* inspected via Accessibility Inspector, *then* every board cell and key control has a unique, stable identifier.

## 8. Data Model

No new production types. The UI test references a copy of the known puzzle solution:

```swift
// UITests/Fixtures/Puzzle001Solution.swift
enum Puzzle001 {
    // Mirror of puzzle-001.json solution; asserted equal to bundled JSON in a unit test.
    static let solution: [(row: Int, col: Int)] = [ /* ...8 positions... */ ]
    static func cellId(row: Int, col: Int) -> String { "cell-\(row)-\(col)" }
}
```

Accessibility identifier contract (production side):

| Element | Identifier |
|---|---|
| Board cell | `cell-<row>-<col>` |
| Win overlay | `win-overlay` |
| Play Again button | `play-again-button` |
| Undo button | `undo-button` |
| Mode toggle | `mode-toggle` |

## 9. API Surface

- No network API. Launch configuration only:

```swift
let app = XCUIApplication()
app.launchArguments += ["-uiTestPuzzle", "puzzle-001", "-uiTestReduceMotion", "1"]
app.launch()

for pos in Puzzle001.solution {
    app.otherElements[Puzzle001.cellId(row: pos.row, col: pos.col)].tap()
}
XCTAssertTrue(app.otherElements["win-overlay"].waitForExistence(timeout: 5))
```

The app MUST read `-uiTestPuzzle` at startup and load that puzzle as the active game (test affordance only; ignored in normal launch).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Flaky taps due to animations | M | M | Disable/relax animations via Reduce Motion launch arg; use `waitForExistence` not sleeps |
| Cell hit-testing ambiguous for XCUITest | M | H | Assign explicit `accessibilityIdentifier` per cell; verify with Accessibility Inspector |
| Test couples to default-puzzle changes | M | M | Force a known puzzle via launch argument |
| Solution fixture drifts from JSON | M | M | Unit test asserts fixture equals bundled `puzzle-001.json` |

## 15. Rollout Plan

- No feature flag. Merge when AC-1–AC-6 pass on the CI simulator.
- CI: add `GameplayUITests` to the required end-to-end check for `main`.
- Rollback: revert the UI test (and identifiers if they cause issues, though they are harmless in production).

## 16. Test Plan

- **End-to-end (primary)** — Launch → place full solution → assert win (AC-1); board locked (AC-2); Play Again resets (AC-4).
- **Interaction** — Undo removes last animal (AC-3).
- **Contract** — Accessibility identifiers present and unique (AC-6); fixture matches bundled JSON.
- **CI** — `xcodebuild test` runs UI target on a pinned simulator (AC-5).
- **Manual exploratory** — Run once on device to confirm identifiers resolve on real hardware.

## 17. Documentation & Training

- Document the `-uiTestPuzzle` launch argument and identifier naming convention in the test README.
- Note how to add UI coverage for future screens using the same identifier pattern.

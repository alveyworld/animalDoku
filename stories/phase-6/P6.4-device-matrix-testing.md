## 1. Problem Statement

Animal Doku's board is a responsive 8×8 grid that must fit and remain fully playable from the smallest supported iPhone (SE) to the largest (Pro Max), and ideally iPad if the target becomes universal. Layout bugs — clipped boards, oversized cells, cramped toolbars, or safe-area collisions — often appear only on specific screen sizes and go unnoticed when developing on one simulator. This story systematically verifies layout and playability across a defined device matrix so no supported device ships broken.

## 2. Goals

- Define and execute a device matrix covering iPhone SE (smallest), a standard iPhone, iPhone Pro Max (largest), and iPad (if universal).
- Verify the board renders square, fully visible, and tappable (≥44pt cells) on every device.
- Verify toolbar, settings sheet, and win overlay layouts adapt without clipping or overlap.
- Confirm portrait-only orientation is enforced per GDD.
- Record results in a device matrix checklist with screenshots as evidence.

## 3. Non-Goals

- Accessibility conformance (P6.3) — separate audit, though layout at large Dynamic Type overlaps and is cross-referenced.
- New adaptive/iPad-specific layouts or split views (only verify current layout adapts; net-new iPad UX is future).
- Performance profiling on older hardware (spot-check only).
- Automated screenshot/snapshot testing infrastructure (optional future).

## 4. Personas & User Stories

- **As a player on an iPhone SE**, I want the whole board visible and tappable so that I can play without scrolling or mis-taps.
- **As a player on a Pro Max**, I want the board centered and appropriately sized so that it doesn't look stretched or lost in whitespace.
- **As an iPad player** (if supported), I want a usable layout so that the app doesn't look like a blurry blown-up phone app.
- **As QA**, I want a documented matrix so that device coverage is repeatable each release.

## 5. Functional Requirements

- **FR-1.** The app MUST be verified on at least: iPhone SE (2nd/3rd gen), iPhone 15/16 (standard), iPhone 15/16 Pro Max, and iPad (if the target is universal).
- **FR-2.** On every device, the 8×8 board MUST render fully visible (no clipping) and square within safe areas.
- **FR-3.** On every device, each cell's tap target MUST remain ≥44×44 pt (or the board scales so the smallest device still meets this, else document the constraint).
- **FR-4.** Toolbar, mode toggle, settings sheet (P4.4), and win overlay (P3.15) MUST render without overlap, clipping, or off-screen elements on every device.
- **FR-5.** The app MUST remain portrait-locked on all devices per GDD (iPad orientation behavior documented).
- **FR-6.** A play-to-win pass (manual or via P6.2 UI test) MUST succeed on at least the smallest and largest iPhone.
- **FR-7.** Results MUST be recorded in a device matrix checklist with a screenshot per device.

## 6. Non-Functional Requirements

- **Performance** — Board scrolling/interaction stays smooth (~60fps) on the oldest supported device (spot-check).
- **Security** — N/A.
- **Privacy** — N/A.
- **Accessibility** — Cross-check that ≥44pt targets (P6.3) hold on the smallest device; note if large Dynamic Type + small screen conflicts.
- **Maintainability** — Layout uses relative/responsive sizing (GeometryReader/aspect ratio) rather than fixed points, verified here.
- **Reliability** — No crashes on any device in the matrix.

## 7. Acceptance Criteria

- **AC-1.** *Given* each device in the matrix, *when* the Game screen loads, *then* the full 8×8 board is visible and square within safe areas.
- **AC-2.** *Given* iPhone SE, *when* measuring a cell, *then* its tap target is ≥44×44 pt (or the documented minimum is met).
- **AC-3.** *Given* each device, *when* opening the settings sheet and triggering the win overlay, *then* no element clips or overlaps.
- **AC-4.** *Given* any device, *when* rotated, *then* the app stays in portrait per GDD.
- **AC-5.** *Given* iPhone SE and Pro Max, *when* a full puzzle is played, *then* it completes to the win screen without layout blockers.
- **AC-6.** *Given* the matrix run, *when* complete, *then* a checklist with per-device screenshots exists and every row is PASS or has a filed defect.

- Must ship after: playable UI (P3.9, P3.14, P3.15), and benefits from the P6.2 UI test and P6.3 audit.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Board clips on iPhone SE | M | M | Use aspect-ratio-constrained board sized to min(width, availableHeight) |
| Excess whitespace on Pro Max/iPad | M | L | Cap board max size; center with padding |
| iPad looks like scaled phone app | M | M | Decide universal vs. iPhone-only; document; defer iPad-specific UX to future |
| Manual matrix skipped under deadline | M | M | Automate play-to-win across destinations in CI to cover the core case |

## 15. Rollout Plan

- Rollback: N/A (verification story); file layout defects against the relevant UI story.

## 16. Test Plan

- **Layout** — Launch on each device; verify board fits/square (AC-1), targets ≥44pt (AC-2), chrome no-clip (AC-3).
- **Orientation** — Attempt rotation; confirm portrait lock (AC-4).
- **End-to-end** — Play-to-win on SE and Pro Max (AC-5), reusing P6.2 where possible.
- **Evidence** — Capture screenshots; fill `docs/device-matrix.md` (AC-6).
- **Manual exploratory** — Tap-accuracy feel check on the smallest device.

## 17. Documentation & Training

- Publish `docs/device-matrix.md` as the per-release device checklist.
- Document the supported-device list and the universal (iPad) decision.

## 18. Open Questions

1. Is the v1.x target iPhone-only or universal (iPad)? **Decided: iPhone-only** (`TARGETED_DEVICE_FAMILY = 1`); iPad deferred.
2. What is the oldest supported device given the iOS 17 floor? **Decided: iPhone SE (3rd gen) / 375 pt as the smallest layout target** (covers SE 2nd-gen class width).

## 1. Problem Statement

The Place / Mark segmented control forces players to constantly switch modes while deducting — mark a few cells, flip to Place, place an animal, flip back. That friction slows the calm loop and caused related bugs when mode and drag state disagreed. Collapsing input into **one tap = mark (X)** and **double tap = place/remove animal** removes the toggle, matches how many logic-puzzle apps feel, and keeps Mark (the frequent action) as the cheap gesture.

## 2. Goals

- Remove the Place / Mark `InputModeToggle` from `GameView` (and retire mode as a player-facing concept).
- **Single tap** (or click) on a cell toggles mark: empty → blocked, blocked → empty; animal → no effect.
- **Double tap** places an animal on empty (or clears mark then places — see Open Questions), or removes an animal if one is present.
- Preserve **drag-to-mark** (P5.7): press-and-drag still paints/clears X marks without requiring a mode.
- Keep VoiceOver / Switch Control playable via explicit custom actions (single-tap/double-tap conflict with VO).
- Update Formal Rules interaction section, tutorial copy (P5.5), and UI tests (P6.2) to the new model.

## 3. Non-Goals

- Drag-to-place animals (still out of scope; drag remains mark-only).
- Triple-tap or long-press alternatives (unless required for a11y fallback — see Open Questions).
- Changing undo/redo semantics beyond recording the resulting place/remove/toggle/markStroke actions.
- Reintroducing a settings preference to restore the old toggle (unless product asks later).

## 4. Personas & User Stories

- **As a player**, I want to mark cells with one tap so that deduction stays fast without flipping a mode switch.
- **As a player**, I want to double-tap to place or remove an animal so that placement is deliberate and harder to do by accident.
- **As a VoiceOver user**, I want clear Mark / Place (or Remove) actions on each cell so that I am not blocked by the system double-tap-to-activate gesture.
- **As a new player**, I want the tutorial to explain tap vs double-tap so that I learn the board without discovering it by mistake.

## 5. Functional Requirements

- **FR-1.** The Place / Mark toggle UI (`InputModeToggle`) MUST be removed from `GameView` and related chrome.
- **FR-2.** `GameSession` / `GameViewModel` MUST stop requiring a player-selected `inputMode` for board taps (mode may be deleted or reduced to an internal implementation detail).
- **FR-3.** **Single tap** on a cell MUST:
  - empty → set `blocked`
  - blocked → set `empty`
  - animal → no effect
- **FR-4.** **Double tap** on a cell MUST:
  - empty → place animal
  - blocked → place animal (MUST clear the mark as part of the place, or reject — **decide in Open Questions**; default recommend: clear + place as one undo step or place-after-clear)
  - animal → remove animal
- **FR-5.** Gesture disambiguation MUST ensure a completed double tap does **not** leave a leftover mark from the first tap (no “flash X then animal”).
- **FR-6.** Drag-to-mark (P5.7) MUST continue to paint/clear marks across cells; starting a drag MUST cancel any pending single-tap timer for that touch.
- **FR-7.** After the puzzle is completed / board locked, taps and double taps MUST no-op (unchanged lock rules).
- **FR-8.** VoiceOver cells MUST expose custom actions (e.g. “Mark” / “Unmark”, “Place animal” / “Remove animal”) equivalent to single/double tap; the default VO activate MAY map to Mark (single-tap) or to a chooser — document the choice.
- **FR-9.** Tutorial (P5.5) and any help copy that mentions Place/Mark modes MUST be updated to tap / double-tap.
- **FR-10.** UI tests (P6.2) MUST place the solution via **double tap** (or the exposed accessibility Place action), not via mode toggle.

## 6. Non-Functional Requirements

- **Performance** — Double-tap window ~250–400 ms; board stays at 60 fps; no perceptible lag on 8×8.
- **Accessibility** — WCAG / Apple guidance: do not rely on double-tap alone; custom actions required. Announce state changes.
- **Reliability** — Pending single-tap work MUST be cancelled on drag start, mode-less teardown, and board lock.
- **Maintainability** — Centralize tap-count / gesture policy in `GameViewModel` or a small `BoardGestureRouter` so `BoardView` stays presentation-focused.
- **Documentation** — Update Formal Rules §Interaction Model to the gesture table; remove “two mutually exclusive input modes” language.

## 7. Acceptance Criteria

- **AC-1.** *Given* an empty cell, *when* the player single-taps, *then* the cell becomes blocked (X).
- **AC-2.** *Given* a blocked cell, *when* the player single-taps, *then* the cell becomes empty.
- **AC-3.** *Given* an empty cell, *when* the player double-taps, *then* an animal is placed and no lasting mark remains from the first tap.
- **AC-4.** *Given* an animal cell, *when* the player double-taps, *then* the animal is removed.
- **AC-5.** *Given* GameView, *when* inspecting the screen, *then* the Place/Mark segmented control is gone.
- **AC-6.** *Given* Mark drag (P5.7), *when* the player drags across empty cells, *then* marks paint as today (one undo stroke).
- **AC-7.** *Given* VoiceOver on a cell, *when* using custom actions, *then* the player can mark and place/remove without needing a mode toggle.
- **AC-8.** *Given* `GameplayUITests`, *when* solving `puzzle-001`, *then* the test completes via the new gesture/action API and the win screen appears.
- **AC-9.** *Given* Formal Rules / tutorial, *when* read after this story, *then* they describe tap / double-tap (not Place/Mark modes).

## 8. Data Model

No persisted schema change. Runtime:

```swift
// Conceptual — replace InputMode-driven tap routing
enum BoardTapKind {
    case single  // mark toggle
    case double  // place / remove
}
```

`GameAction` history remains place / remove / toggleBlocked / markStroke / hint. Optional: one combined action if double-tap on blocked clears+places atomically.

`InputMode` enum and `session.inputMode` SHOULD be removed once call sites are gone (or left unused briefly behind a deprecate comment — prefer delete).

## 9. API Surface

N/A (app-internal). Suggested view-model surface:

```swift
func handleCellSingleTap(at position: Position)
func handleCellDoubleTap(at position: Position)
// Drag API from P5.7 unchanged: beginMarkDrag / continueMarkDrag / endMarkDrag
```

`BoardView` detects tap count (or uses `onTapGesture(count:)` + delayed single) and calls the above.

## 10. UI / UX

- Remove `InputModeToggle` and its caption (“Tap cells to place…” / mark hint).
- Optional short hint under the board once: “Tap to mark · Double-tap to place” (dismissible or tutorial-only).
- Visual feedback unchanged (X mark, animal icon, selection).
- Double-tap timing should feel forgiving on trackpad/mouse (Mac Catalyst / pointer) if supported; use the same window.
- States: idle → pendingSingleTap → (fires mark | upgraded to double → place).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Single tap always fires before double tap | H | H | Delay single-tap commit until double-tap timeout; cancel on second tap |
| Double-tap feels slow / laggy marks | M | M | Tune window (~300 ms); optional immediate mark + revert if double arrives |
| VoiceOver double-tap conflict | H | H | Custom actions; don’t require physical double-tap for VO |
| Accidental marks while aiming to place | M | M | Teach in tutorial; Undo; forgiving double-tap window |
| P5.7 drag + tap timer fights | M | M | Drag began cancels pending single tap |
| Players miss “double-tap to place” | M | M | Tutorial step + optional on-board hint |

## 18. Open Questions

1. **Double-tap on blocked:** Clear + place, or no effect until single-tap clears? **Decided: clear + place** (`place` with `previous: .blocked`).
2. **Single-tap timing:** Delay-commit vs optimistic mark-then-revert on double? **Decided: SwiftUI `onTapGesture(count:)`** (double registered before single).
3. **Single-tap on animal:** **Decided: no effect; remove only on double tap.**
4. **On-board hint strip:** **Decided: always-visible caption** under the title (“Tap to mark · Double-tap to place”).
5. **VoiceOver:** Default custom actions “Mark” / “Clear mark” and “Place animal” / “Remove animal” on each cell.

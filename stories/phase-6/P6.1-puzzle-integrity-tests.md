## 1. Problem Statement

Every bundled puzzle must satisfy hard invariants — regions partition all `size × size` cells exactly once, the declared `solution` obeys all four rules, and exactly one solution exists. If a malformed puzzle ships, players can hit an unsolvable board, a region with no valid placement, or a win that never triggers. Manual verification does not scale as content grows from one puzzle (v0.1) to 5–10 (v1.0), so an automated integrity suite must gate every puzzle JSON before it ships.

## 2. Goals

- Add `Tests/PuzzleIntegrityTests.swift` that loads every bundled puzzle and asserts the [Formal Rules invariants](../../AnimalDoku_Formal_Rules_and_Data_Model.md#invariants).
- Prove each puzzle's `solution` is valid and complete via the real `Validator` (P2.7).
- Verify structural invariants that the validator does not check: region partition, contiguous region IDs, unique colors/patterns, `solution.length == size`, empty `initialPlacements`.
- Provide a single-solution check for the MVP puzzle(s) via a brute-force/backtracking solver used in tests only.
- Make the suite data-driven so a newly added JSON is automatically covered without editing test code.

## 3. Non-Goals

- Validator rule-logic unit tests (owned by P2.9).
- `GameSession` / undo-redo tests (P3.7).
- UI or accessibility tests (P6.2, P6.3).
- A runtime solver in the shipping app — solving/verification is offline/test-time only per Formal Rules §Content Verification.
- Authoring new puzzles (P2.8 for MVP; content expansion is separate).

## 4. Personas & User Stories

- **As a puzzle author**, I want automated invariant checks so that a malformed puzzle fails CI instead of reaching players.
- **As a developer**, I want a single-solution guarantee so that hints and win detection are trustworthy.
- **As a player** (indirect), I want every puzzle to be fair and solvable so that the game feels polished, not broken.

## 5. Functional Requirements

- **FR-1.** `Tests/PuzzleIntegrityTests.swift` MUST exist in the unit test target and load all puzzles from `Resources/Puzzles/` via `PuzzleLoader` (P2.3).
- **FR-2.** For every puzzle, tests MUST assert regions partition all `size × size` cells exactly once — no cell uncovered, no cell in two regions (Formal Rules invariant 2).
- **FR-3.** For every puzzle, region `id` values MUST be contiguous `0…size-1` (invariant 3).
- **FR-4.** For every puzzle, each region MUST have a unique `color` and a distinct pattern index (invariant 4).
- **FR-5.** For every puzzle, `solution.count == size` MUST hold (invariant 5).
- **FR-6.** For every puzzle, building a `Board` from `solution` and running `Validator.validate` MUST return `isValid == true`, `isComplete == true`, empty `violations` (invariants 6, Rule 5).
- **FR-7.** For every puzzle, `initialPlacements` MUST be `[]` for MVP (invariant 8).
- **FR-8.** The suite MUST include a test-only solver that confirms the declared `solution` is the **only** solution for at least the MVP puzzle(s) (invariant 7).
- **FR-9.** The suite MUST fail with a clear message identifying the offending puzzle `id` and the violated invariant.
- **FR-10.** Adding a new puzzle JSON MUST be covered automatically (iterate `loadAllPuzzles()`), requiring no new test method.

## 6. Non-Functional Requirements

- **Performance** — Partition/validator checks run in <1s total; the single-solution solver on an 8×8 board MUST complete in a few seconds (backtracking with pruning on rows).
- **Security** — No network; reads only bundled resources.
- **Privacy** — N/A.
- **Accessibility** — N/A (logic tests). Pattern-index uniqueness (FR-4) supports the colorblind requirement enforced elsewhere.
- **Maintainability** — Data-driven; one parametrized path over all puzzles. Solver lives in the test target only, never in the app.
- **Determinism** — No randomness; stable iteration order by puzzle `id`.

## 7. Acceptance Criteria

- **AC-1.** *Given* all bundled puzzles, *when* the suite runs, *then* every puzzle's regions cover all `size × size` cells exactly once or the test fails naming the puzzle.
- **AC-2.** *Given* a puzzle, *when* region IDs are inspected, *then* they are exactly `0…size-1` with no gaps or duplicates.
- **AC-3.** *Given* a puzzle's `solution`, *when* validated, *then* `isValid` and `isComplete` are both `true` and `violations` is empty.
- **AC-4.** *Given* a puzzle, *when* checked, *then* `solution.count == size` and `initialPlacements` is empty.
- **AC-5.** *Given* the MVP puzzle, *when* the test solver enumerates solutions, *then* exactly one solution exists and it equals `puzzle.solution` (as a set of positions).
- **AC-6.** *Given* an intentionally broken fixture (region overlap or extra solution), *when* the suite runs, *then* the corresponding test fails with a descriptive message.
- **AC-7.** *Given* `xcodebuild test`, *when* run, *then* all `PuzzleIntegrityTests` pass for the shipped content set.

## 8. Data Model

No new production types. Test-only helper:

```swift
// Tests/Helpers/PuzzleSolver.swift (test target only)
enum PuzzleSolver {
    /// Enumerates all valid solutions (one animal per row, obeying Rules 1–4).
    /// Returns early once a second solution is found for uniqueness checks.
    static func solutions(for puzzle: Puzzle, limit: Int = 2) -> [[Position]]
}
```

Invariants under test (from [Formal Rules §Invariants](../../AnimalDoku_Formal_Rules_and_Data_Model.md#invariants)):

| # | Invariant | Checked by |
|---|---|---|
| 2 | Regions partition all cells once | FR-2 / AC-1 |
| 3 | Region IDs contiguous `0…size-1` | FR-3 / AC-2 |
| 4 | Unique color + pattern index | FR-4 |
| 5 | `solution.length == size` | FR-5 / AC-4 |
| 6 | `solution` satisfies Rules 1–4 | FR-6 / AC-3 |
| 7 | `solution` is the only solution | FR-8 / AC-5 |
| 8 | `initialPlacements == []` | FR-7 / AC-4 |

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Solver too slow for larger/expert boards | M | M | Prune by row + no-touch early; cap `limit` at 2; keep MVP at 8×8 |
| Pattern-index field not yet modeled | M | M | Coordinate with P1.5/P2.1; assert on `color` uniqueness now, add pattern check when field exists |
| False negatives from off-by-one region math | L | M | Unit-test the partition helper against a known-good fixture |
| Broken-fixture test pollutes bundle | L | M | Keep bad fixtures in test resources, not app bundle |

## 15. Rollout Plan

- No feature flag. Merge when AC-1–AC-7 pass on shipped content.
- CI: add `PuzzleIntegrityTests` to required checks; every new puzzle PR must pass.
- Rollback: revert test file only; no production impact.

## 16. Test Plan

This story **is** a test suite. Organize `PuzzleIntegrityTests.swift` into sections:

- **Structure** — region partition (AC-1), contiguous IDs (AC-2), unique color/pattern (FR-4), `solution.count`, empty `initialPlacements` (AC-4).
- **Solution validity** — validate each `solution` board → valid + complete (AC-3).
- **Uniqueness** — solver enumerates ≤2 solutions on MVP puzzle; assert exactly one equals declared solution (AC-5).
- **Negative fixtures** — overlapping regions and a second-solution puzzle must fail (AC-6).

**Manual exploratory:** None beyond confirming `Cmd+U` passes and failure messages name the puzzle.

## 17. Documentation & Training

## 18. Open Questions

1. Does the `Region`/puzzle schema carry an explicit `patternIndex`, or is pattern derived from region `id`? Recommend deriving from `id` for MVP and asserting color uniqueness now.
2. Should the uniqueness solver run on all puzzles or only newly added ones in CI (for speed)? Recommend all at 8×8 (fast enough); revisit if expert boards grow.

## 1. Problem Statement

The board still uses soft pastel region fills (from design tokens / puzzle JSON) plus hatch/dot **pattern overlays** (P3.10) so regions stay distinguishable under colorblindness. The new look-and-feel direction is **brighter, higher-contrast solid colors** with **no pattern lines** — relying on a curated accessible palette instead of overlays. Until the palette and pattern policy change, the UI won’t match the intended visual identity or the provided accessibility color set.

## 2. Goals

- Adopt the provided accessible palette as the **canonical region fill set** (and align chrome/accent tokens where it improves contrast).
- Render regions as **solid fills only** — remove (or disable) P3.10 pattern line overlays on the board.
- Keep regions distinguishable for common color-vision deficiencies via palette choice + cell borders (not hatch lines).
- Preserve High Contrast mode (P5.3) as a further-boosted path, using the same palette family (or a documented subset).
- Update tests and the accessibility audit notes so “patterns required” is no longer assumed for default play.

## 3. Non-Goals

- Redesigning animal theme icons or app icon (separate stories if needed).
- Dark mode.
- Changing puzzle rules, layouts, or solution data (only presentation / region hex if we rewrite JSON).
- Reintroducing a “show patterns” settings toggle (unless product asks later).
- Full App Store / TestFlight work (Phase 7).

## 4. Personas & User Stories

- **As a player**, I want bright, clear region colors so that the board feels vibrant and easy to scan.
- **As a colorblind player**, I want regions to stay distinguishable without relying on faint pastels plus busy hatch lines.
- **As a developer**, I want one named palette in the design system so puzzles and UI don’t drift into one-off hex values.

## 5. Functional Requirements

- **FR-1.** The design system MUST define the accessible palette as named tokens (see §8).
- **FR-2.** Board region fills MUST use this palette (by `regionId` order / modulo), not the current pastel `AppColors.regionPalette` alone.
- **FR-3.** `BoardView` MUST **not** draw `RegionPattern` overlays in the default and high-contrast board presentation for this look (“no lines”).
- **FR-4.** Cell grid borders / selection / violation chrome MUST remain so the 8×8 structure stays clear without pattern lines.
- **FR-5.** Text and icon contrast on backgrounds MUST remain ≥ 4.5:1 for primary UI chrome (retune `primary` / `secondary` / `accent` if the new fills force it).
- **FR-6.** Puzzle JSON `region.color` values SHOULD either be updated to match the palette **or** be ignored at render time in favor of the token palette (document the choice; prefer token override for one source of truth).
- **FR-7.** Unit tests MUST assert palette hex values and that region pattern overlays are not required for board rendering in this mode.
- **FR-8.** `docs/accessibility-audit.md` MUST be updated to reflect solid-color regions (no default hatch patterns).

## 6. Non-Functional Requirements

- **Performance** — Removing pattern `Canvas` overlays should be neutral or faster.
- **Accessibility** — Palette chosen for distinguishability; WCAG AA for UI text/icons; VoiceOver region labels unchanged (still “Region N”).
- **Maintainability** — Single `AccessiblePalette` (or equivalent) in `Colors.swift`; board reads tokens only.
- **Reliability** — Themes (Frogs/Dogs/Foxes) animal icons still readable on the brighter fills.

## 7. Acceptance Criteria

- **AC-1.** *Given* the game board, *when* regions render, *then* fills use the accessible palette colors (not pastels).
- **AC-2.** *Given* the game board, *when* inspecting cells, *then* there are **no** hatch/dot/line pattern overlays on region fills.
- **AC-3.** *Given* an 8-region puzzle, *when* each region is sampled, *then* region ids map stably to distinct palette entries.
- **AC-4.** *Given* High Contrast mode, *when* enabled, *then* regions remain solid (no patterns) and still use the accessible / high-contrast palette path.
- **AC-5.** *Given* primary text on the screen background, *when* contrast is measured, *then* ratio is ≥ 4.5:1.
- **AC-6.** *Given* `RegionPattern` / board tests, *when* run after this story, *then* they match the no-overlay policy (update or retire obsolete assertions).

## 8. Data Model

No new persisted types. Design tokens:

```text
Accessible region / accent palette (authoritative):
Blue              #0072B2
Orange            #E69F00
Green             #009E73
Vermillion        #D55E00
Sky Blue          #56B4E9
Reddish Purple    #CC79A7
Yellow            #F0E442
Dark Gray         #4D4D4D
Bright Pink       #FF4DA6
Teal              #00BFC4
Olive             #7CAE00
Violet            #8B5CF6
```

Suggested mapping for 8×8 MVP regions (ids 0…7) — first eight, stable order:

| regionId | Name | Hex |
|---------:|------|-----|
| 0 | Blue | `#0072B2` |
| 1 | Orange | `#E69F00` |
| 2 | Green | `#009E73` |
| 3 | Vermillion | `#D55E00` |
| 4 | Sky Blue | `#56B4E9` |
| 5 | Reddish Purple | `#CC79A7` |
| 6 | Yellow | `#F0E442` |
| 7 | Dark Gray | `#4D4D4D` |

Reserve Bright Pink / Teal / Olive / Violet for larger boards or future accents (toolbar, win, theme accents).

```swift
// Conceptual — UI/DesignSystem/Colors.swift

        static let blue = Color(hex: "#0072B2")

}
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Yellow / sky fills wash out animal icons | M | M | Use dark icon/primary on light fills; verify each region |
| Colorblind users lose hatch cues | M | H | Rely on Okabe–Ito–style palette + borders; spot-check deuteranopia simulation |
| Puzzle JSON hex fights token palette | M | L | Override at render by `regionId` (single source of truth) |
| High Contrast mode + bright fills clash | L | M | Define HC as same hues / darker borders / white bg |

## 15. Rollout Plan

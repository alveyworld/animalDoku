## 1. Problem Statement

The core of the game is Rules 1–3: exactly one animal per row, column, and colored region. Nothing can enforce these rules today. The `Validator` engine that checks uniqueness is the foundation for adjacency, completion, and violation feedback, and for wiring placement through validation in gameplay. This story establishes that foundation.

## 2. Goals

- Create `Engine/Validator.swift` with a function that evaluates a board's cells against Rules 1–3.
- Detect when a row, column, or region contains more than one animal.
- Return enough information to identify which line/region is violated and the offending positions.
- Keep the validator pure (no state, no SwiftUI) and O(n) per pass, where n = `size`.

## 3. Non-Goals

- No-touch adjacency (Rule 4 → P2.5).
- Completion check (Rule 5 → P2.6).
- The full `ValidationResult`/`RuleViolation` public shape (P2.7 formalizes and unifies it; this story may return an internal list first).
- Applying validation results to UI highlighting.

## 4. Personas & User Stories

- **As a developer**, I want a pure uniqueness check so that I can compose it into the full validator.
- **As a player** (indirect), I want the game to recognize when I place two animals in the same row so that mistakes are visible.
- **As a QA engineer**, I want deterministic violation output so that tests assert exact positions.

## 5. Functional Requirements

- **FR-1.** The `Validator` MUST accept the current board (cells with `state` and `regionId`) and the puzzle/region layout.
- **FR-2.** The `Validator` MUST detect a **row** violation when two or more animals share the same `row`.
- **FR-3.** The `Validator` MUST detect a **column** violation when two or more animals share the same `col`.
- **FR-4.** The `Validator` MUST detect a **region** violation when two or more animals share the same `regionId`.
- **FR-5.** A violation MUST include the rule type (`row`, `column`, or `region`) and the list of positions of the animals involved in that line/region.
- **FR-6.** With zero animals or exactly one animal per line/region, the uniqueness check MUST report no violations.
- **FR-7.** The `Validator` MUST be a pure function (same input → same output; no side effects) and MUST NOT import SwiftUI/UIKit.

## 6. Non-Functional Requirements

- **Performance** — Uniqueness pass MUST be O(n) in `size` (bucket by row/col/region), completing in well under 1 ms for 8×8.
- **Reliability** — Deterministic ordering of returned positions (e.g. row-major) so tests are stable.
- **Maintainability** — Single validator file; row/col/region logic factored into small private helpers.
- **Accessibility** — N/A (engine only; feeds UI later).

## 7. Acceptance Criteria

- **AC-1.** *Given* two animals in the same row (per Formal Rules Rule 1 "Invalid" example), *when* validated, *then* a `row` violation listing both positions is returned. *(maps to Formal Rules AC-V.1)*
- **AC-2.** *Given* two animals in the same column, *when* validated, *then* a `column` violation listing both positions is returned.
- **AC-3.** *Given* two animals in the same region, *when* validated, *then* a `region` violation listing both positions is returned.
- **AC-4.** *Given* a board with at most one animal per row, column, and region, *when* validated, *then* no uniqueness violation is returned.
- **AC-5.** *Given* an empty board, *when* validated, *then* no violations are returned.

## 8. Data Model

- Consumes `Cell`, `Position`, `Region`, `Puzzle` from P2.1.
- Uses the `RuleType`/`RuleViolation` concepts from Formal Rules §Validation. This story may introduce a minimal internal form; P2.7 finalizes the public `ValidationResult`:

```swift
enum RuleType: String, Codable { case row, column, region, adjacency }

struct RuleViolation: Equatable {
    let rule: RuleType
    let positions: [Position]
}
```

## 9. API Surface

- No network APIs.
- Internal engine surface (extended in later stories):

```swift
struct Validator {
    // Rules 1–3 in this story; adjacency/completion added in P2.5/P2.6.
    func uniquenessViolations(cells: [Cell], size: Int) -> [RuleViolation]
}
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Off-by-one in row/col bucketing | M | H | Unit tests from Formal Rules examples (AC-1..AC-3) |
| Region violation missed for irregular shapes | M | H | Bucket by `regionId` directly, independent of shape |
| Non-deterministic position order | L | M | Sort violation positions row-major |

## 1. Problem Statement

The individual rule checks (uniqueness, no-touch, completion) exist, but there is no single, structured result the rest of the app can consume. Per the Formal Rules interaction model, invalid placements are **allowed but highlighted**, so the UI needs a typed list of violations and affected positions on every board change. This story unifies the checks into one `ValidationResult` and defines `RuleViolation`/`RuleType`.

## 2. Goals

- Define the public `ValidationResult`, `RuleViolation`, and `RuleType` types per Formal Rules Â§Validation.
- Provide a single `Validator.validate(...)` entry point that runs Rules 1â€“4, aggregates violations, and computes `isValid`/`isComplete`.
- Support the "allow invalid placement, highlight it" behavior by returning affected positions for every violation.
- Ensure results are `Equatable` so tests and view models can diff them.

## 3. Non-Goals

- Rendering highlights (Phase 3 CellView/BoardView consume this).
- Blocking or rejecting invalid placements (product decision is to allow + highlight, per Formal Rules).
- Undo/redo or session state (Phase 3).

## 4. Personas & User Stories

- **As a player**, I want invalid placements to stay on the board but be visibly flagged so that I can reason about my mistake without being blocked.
- **As a developer**, I want one `validate` call returning a complete result so that view models don't orchestrate multiple checks.
- **As a QA engineer**, I want `Equatable` results so that I can assert the exact violation set.

## 5. Functional Requirements

- **FR-1.** The system MUST define `RuleType` (`row`, `column`, `region`, `adjacency`), `RuleViolation { rule, positions }`, and `ValidationResult { isValid, isComplete, violations }` exactly per Formal Rules Â§Validation.
- **FR-2.** The `Validator` MUST expose a single `validate(cells:puzzle:)` (or equivalent) that returns a `ValidationResult`.
- **FR-3.** `validate` MUST aggregate all violations from Rules 1â€“4 into `violations` (not stop at the first).
- **FR-4.** `isValid` MUST be `true` iff `violations` is empty.
- **FR-5.** `isComplete` MUST be `true` iff `isValid` and Rule 5 completion holds (from P2.6).
- **FR-6.** Invalid placements MUST NOT be rejected by the validator; the validator only reports â€” placement/highlight policy stays "allow + highlight" per Formal Rules Â§Placement Validation.
- **FR-7.** `RuleViolation` and `ValidationResult` MUST be `Equatable`; `RuleType`/`RuleViolation` `Codable` where the spec shows it.
- **FR-8.** The validator MUST remain pure and free of SwiftUI/UIKit.

## 6. Non-Functional Requirements

- **Performance** â€” One `validate` pass MUST be O(n) in `size` and run on every board mutation without perceptible lag (< 1 ms for 8Ă—8).
- **Reliability** â€” Deterministic violation ordering (e.g. by rule type, then row-major positions) so UI and tests are stable.
- **Maintainability** â€” `ValidationResult` and friends in `Engine/ValidationResult.swift`; `validate` in `Engine/Validator.swift`.
- **Accessibility** â€” Violation positions feed colorblind-safe highlight + VoiceOver hints downstream (Phase 5); design result to carry positions per violation.

## 7. Acceptance Criteria

- **AC-1.** *Given* two animals in the same row, *when* `validate` is called, *then* `isValid` is `false` and a `row` violation lists both positions. *(Formal Rules AC-V.1)*
- **AC-2.** *Given* two diagonally adjacent animals, *when* `validate` is called, *then* an `adjacency` violation is returned. *(Formal Rules AC-V.2)*
- **AC-3.** *Given* a board matching `solution`, *when* `validate` is called, *then* `isValid` and `isComplete` are both `true`. *(Formal Rules AC-V.3)*
- **AC-4.** *Given* a board with multiple simultaneous violations (e.g. a row conflict and an adjacency), *when* `validate` is called, *then* `violations` contains all of them.
- **AC-5.** *Given* an invalid placement, *when* `validate` is called, *then* the offending animals remain in `cells` (the validator does not mutate or reject) and are reported in `violations`.
- **AC-6.** *Given* two `ValidationResult` values with identical violations, *when* compared, *then* they are `Equatable`-equal.

## 8. Data Model

Public engine types (per Formal Rules Â§Validation Â§ValidationResult):

```swift
enum RuleType: String, Codable { case row, column, region, adjacency }

struct RuleViolation: Equatable {
    let rule: RuleType
    let positions: [Position]
}

struct ValidationResult: Equatable {
    let isValid: Bool        // true if zero violations
    let isComplete: Bool     // true if valid and Rule 5 satisfied
    let violations: [RuleViolation]
}
```

- `positions` carries every cell to highlight for that violation.

## 1. Problem Statement

Rule 4 forbids animals from touching in any of the eight surrounding cells, including diagonals. The uniqueness checks (P2.4) alone will accept illegal boards where animals sit adjacent. Adjacency detection is required before completion can be judged and before the UI can highlight invalid placements. This story adds the no-touch check to the `Validator`.

## 2. Goals

- Extend `Engine/Validator.swift` to detect any two animals adjacent in the 8 directions (N, S, E, W, NE, NW, SE, SW).
- Correctly handle board edges and corners (fewer neighbors, no wrap-around, no out-of-bounds).
- Return an `adjacency` violation identifying the touching animal positions.
- Keep the check pure and O(n) in `size`.

## 3. Non-Goals

- Uniqueness checks (P2.4, already done).
- Completion (P2.6).
- Unifying the result type (P2.7).
- Any UI rendering of adjacency.

## 4. Personas & User Stories

- **As a player**, I want the game to flag animals placed next to each other (even diagonally) so that I learn the no-touch rule.
- **As a developer**, I want a reusable adjacency check so that completion logic can rely on it.
- **As a QA engineer**, I want edge/corner cases covered so that boundary bugs are caught.

## 5. Functional Requirements

- **FR-1.** The `Validator` MUST report an `adjacency` violation when two animals occupy any of the 8 neighboring cells of one another.
- **FR-2.** Adjacency MUST include diagonals (NE, NW, SE, SW), not just orthogonal neighbors.
- **FR-3.** The check MUST NOT wrap around edges (cell at col 0 is not adjacent to col `size-1`).
- **FR-4.** The check MUST safely handle edge and corner cells (only in-bounds neighbors considered).
- **FR-5.** An adjacency violation MUST list the positions of the touching animals.
- **FR-6.** With no two animals touching, the adjacency check MUST report no adjacency violation.
- **FR-7.** The check MUST remain a pure function with no SwiftUI/UIKit imports.

## 6. Non-Functional Requirements

- **Performance** — Adjacency pass MUST be O(n) in `size` (iterate placed animals, check ≤8 offsets each), sub-millisecond for 8×8.
- **Reliability** — Deterministic pairing/ordering of reported positions for stable tests.
- **Maintainability** — Reuse a shared set of 8 direction offsets; no duplicated neighbor logic.

## 7. Acceptance Criteria

- **AC-1.** *Given* two diagonally adjacent animals, *when* validated, *then* an `adjacency` violation is returned. *(maps to Formal Rules AC-V.2)*
- **AC-2.** *Given* two orthogonally adjacent animals (N/S/E/W), *when* validated, *then* an `adjacency` violation is returned.
- **AC-3.** *Given* two animals separated by at least one cell in every direction, *when* validated, *then* no adjacency violation is returned.
- **AC-4.** *Given* an animal in a corner (e.g. row 0, col 0) with a neighbor at (0,1) or (1,1), *when* validated, *then* an adjacency violation is returned and no out-of-bounds access occurs.
- **AC-5.** *Given* animals at opposite edges of the same row (col 0 and col `size-1`), *when* validated, *then* no adjacency violation is returned (no wrap-around).

## 8. Data Model

- Reuses `RuleType.adjacency` and `RuleViolation` (from P2.4 / Formal Rules §Validation). No new types.
- Direction offsets:

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Out-of-bounds crash at edges | M | H | Bounds-check every offset; corner tests (AC-4) |
| Diagonals forgotten | M | H | Explicit 8-offset table; AC-1 asserts diagonal |
| Duplicate violations for same pair | L | L | Dedupe pairs (e.g. only report when neighbor is "after" in row-major) |

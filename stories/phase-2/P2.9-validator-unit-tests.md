## 1. Problem Statement

The `Validator` is the correctness core of the game — every placement is judged by it. Bugs here silently produce false wins or false violations that erode player trust. A comprehensive unit test suite codifies the Formal Rules acceptance criteria (AC-V.x, AC-5.x) as executable checks and guards against regressions as gameplay is built on top.

## 2. Goals

- Create `Tests/ValidatorTests.swift` covering Rules 1–4, completion (Rule 5), and structured violations.
- Encode the Formal Rules acceptance criteria (AC-V.1–V.3, AC-5.1–5.3) as tests.
- Cover edge cases: empty board, single animal, board edges/corners, multiple simultaneous violations.
- Ensure the suite is fast, deterministic, and independent of UI.

## 3. Non-Goals

- GameSession / undo-redo tests (Phase 3, T-081).
- UI or end-to-end tests (Phase 6).
- Puzzle-integrity tests for authored content (covered in P2.8 / Phase 6, though may share fixtures).

## 4. Personas & User Stories

- **As a developer**, I want a validator test suite so that I can refactor the engine without fear of regressions.
- **As a QA engineer**, I want each Formal Rules AC mapped to a test so that spec compliance is provable.
- **As a reviewer**, I want failing tests on rule bugs so that broken logic never merges.

## 5. Functional Requirements

- **FR-1.** The suite MUST live in `Tests/ValidatorTests.swift` and run under the project's test target (from P1.3).
- **FR-2.** The suite MUST include a test for each Formal Rules acceptance criterion: AC-V.1 (row violation), AC-V.2 (adjacency), AC-V.3 (valid+complete solution), AC-5.1 (partial incomplete), AC-5.2 (solution complete), AC-5.3 (full invalid incomplete).
- **FR-3.** The suite MUST cover column and region uniqueness violations (not only row).
- **FR-4.** The suite MUST cover adjacency edge/corner cases and the no-wrap-around case.
- **FR-5.** The suite MUST cover multiple simultaneous violations aggregated into one `ValidationResult`.
- **FR-6.** The suite MUST use small deterministic fixtures (helpers to build a board of `Cell`s from placements) and assert exact `RuleViolation` sets where practical.
- **FR-7.** Tests MUST NOT depend on SwiftUI, timing, randomness, or the file system beyond loading a test puzzle fixture.

## 6. Non-Functional Requirements

- **Performance** — Full suite MUST run in < 2 seconds locally and in CI.
- **Reliability** — 100% deterministic; no flakiness. Fixed fixtures, sorted assertions.
- **Maintainability** — Shared board-builder test helpers; one test method per behavior with descriptive names.
- **Observability** — Failures MUST clearly report expected vs. actual violation sets.

## 7. Acceptance Criteria

- **AC-1.** *Given* two animals in one row, *when* validated, *then* the test asserts `isValid == false` and a `row` violation with both positions. *(Formal Rules AC-V.1)*
- **AC-2.** *Given* two diagonally adjacent animals, *when* validated, *then* the test asserts an `adjacency` violation. *(Formal Rules AC-V.2)*
- **AC-3.** *Given* a board matching a fixture `solution`, *when* validated, *then* the test asserts `isValid` and `isComplete` are both `true`. *(Formal Rules AC-V.3, AC-5.2)*
- **AC-4.** *Given* a partial board, *when* validated, *then* the test asserts `isComplete == false`. *(Formal Rules AC-5.1)*
- **AC-5.** *Given* a full board with a violation, *when* validated, *then* the test asserts `isComplete == false`. *(Formal Rules AC-5.3)*
- **AC-6.** *Given* column-conflict, region-conflict, corner-adjacency, no-wrap, and multi-violation fixtures, *when* validated, *then* each corresponding test passes.
- **AC-7.** *Given* the whole suite, *when* executed in CI, *then* all tests pass and complete in < 2s.

## 8. Data Model

- No production types. Introduces test helpers, e.g.:

```swift
// Build a board where the given positions are `.animal`, rest `.empty`,
// using a region map (regionId per cell) from a fixture puzzle.
func makeBoard(size: Int, regionMap: [[Int]], animals: [Position]) -> [Cell]
```

- May reuse `puzzle-001.json` (P2.8) and small synthetic layouts (e.g. 4×4) as fixtures.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Tests assert wrong "expected" and mask bugs | M | H | Derive expectations from Formal Rules examples verbatim |
| Flaky due to unordered violations | M | M | Sort/compare as sets; deterministic fixtures |
| Coverage gaps (e.g. region for irregular shapes) | M | M | Explicit region-conflict fixture with irregular region |

## 15. Rollout Plan

- No feature flag. Merge when the suite is green.
- Rollback: test-only; safe to revert. Should instead be fixed forward if failing.

## 16. Test Plan

- **Unit** — This story *is* the unit test plan. Test methods: rowViolation, columnViolation, regionViolation, diagonalAdjacency, orthogonalAdjacency, cornerAdjacency, noWrapAround, partialIncomplete, solutionComplete, fullInvalidIncomplete, multipleViolationsAggregated, emptyBoardClean, singleAnimalClean.
- **CI** — Suite runs on every PR; must pass (AC-7).
- **Manual** — Run once locally via `Cmd-U` / `xcodebuild test` to confirm green.

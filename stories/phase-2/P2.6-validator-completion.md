## 1. Problem Statement

The game must know when a puzzle is solved so it can trigger the win state and lock the board. Rule 5 defines completion as: exactly one animal per row, column, and region, no animals touching, and no remaining violations. Uniqueness (P2.4) and no-touch (P2.5) exist, but nothing combines them into a completion verdict. This story adds that check.

## 2. Goals

- Add a completion determination to `Engine/Validator.swift` that returns `isComplete`.
- Define completion strictly per Rule 5: valid (zero violations) AND exactly `size` animals placed, one in each row, column, and region.
- Ensure partial boards and full-but-invalid boards report `isComplete == false`.
- Keep the check pure and O(n).

## 3. Non-Goals

- Triggering UI or win animations (Phase 3/5).
- The full structured `ValidationResult` API (P2.7 formalizes and returns it as a unit).
- Locking the board / play-again (Phase 3 GameSession).

## 4. Personas & User Stories

- **As a player**, I want the game to recognize when I've solved the puzzle so that I get a win screen.
- **As a developer**, I want a single `isComplete` verdict so that the session can transition to the won state.
- **As a QA engineer**, I want completion to be false on any violation so that false wins are impossible.

## 5. Functional Requirements

- **FR-1.** The `Validator` MUST report `isComplete == true` only when there are zero rule violations (Rules 1–4 all satisfied).
- **FR-2.** The `Validator` MUST require exactly `size` animals placed, with exactly one per row, one per column, and one per region.
- **FR-3.** The `Validator` MUST report `isComplete == false` for any partial board (fewer than `size` animals).
- **FR-4.** The `Validator` MUST report `isComplete == false` for a full board that contains any violation (uniqueness or adjacency).
- **FR-5.** Completion MUST be derivable purely from the board `cells` and `size`; it MUST NOT require comparing against `puzzle.solution` (a valid full board is complete by rule). *(Note: since the puzzle has a unique solution, a rule-valid full board equals the solution.)*
- **FR-6.** The check MUST be pure and free of SwiftUI/UIKit.

## 6. Non-Functional Requirements

- **Performance** — Completion determination MUST be O(n) in `size`, reusing the uniqueness/adjacency passes.
- **Reliability** — No false positives: a single violation anywhere yields `isComplete == false`.
- **Maintainability** — Compose existing checks rather than reimplementing rule logic.

## 7. Acceptance Criteria

- **AC-1.** *Given* a partial board, *when* validated, *then* `isComplete` is `false`. *(Formal Rules AC-5.1)*
- **AC-2.** *Given* a board matching `puzzle.solution`, *when* validated, *then* `isComplete` is `true`. *(Formal Rules AC-5.2, AC-V.3)*
- **AC-3.** *Given* a full board (size animals placed) with any rule violation, *when* validated, *then* `isComplete` is `false`. *(Formal Rules AC-5.3)*
- **AC-4.** *Given* a full valid board, *when* validated, *then* `isValid` is `true` and `isComplete` is `true` together.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| False win on full invalid board | M | H | Gate completion on `isValid`; AC-3 covers it |
| Completion true with < size animals | L | H | Require `animalCount == size`; AC-1 covers it |
| Divergence from `solution` uniqueness assumption | L | M | Rely on puzzle single-solution invariant (Formal Rules §Invariants) |

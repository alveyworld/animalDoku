## 1. Problem Statement

Players make experimental placements and marks while solving; without undo they must manually reverse mistakes, which breaks the "calm, self-paced" feel. `GameSession` records actions (P3.2) but cannot yet reverse them. Undo is a core MVP player action per Formal Rules.

## 2. Goals

- Reverse the last recorded action, restoring the exact prior board state.
- Move the reversed action onto the redo stack (consumed by P3.4).
- Support repeated undos back to the initial empty board.
- Make undo a safe no-op when history is empty.

## 3. Non-Goals

- Redo (P3.4) — this story only produces the redo entries.
- Reset (P3.5) — clearing entire history is separate.
- Toolbar button / enabled state UI (P3.12).

## 4. Personas & User Stories

- **As a player**, I want to undo my last move so I can safely experiment while solving.
- **As a player**, I want repeated undos to walk back the whole attempt to the empty board.
- **As a developer**, I want undo built on the same `GameAction` records so behavior is consistent with redo.

## 5. Functional Requirements

- **FR-1.** `undo()` MUST pop the last action from `undoStack` and restore the board to the state prior to that action.
- **FR-2.** The popped action MUST be pushed onto `redoStack`.
- **FR-3.** `undo()` on an empty `undoStack` MUST be a no-op (no crash, no state change).
- **FR-4.** Undo MUST correctly reverse every action type: `placeAnimal` → cell empty; `removeAnimal` → cell animal; `toggleBlocked` → previous state; `hint` → cell empty **and** decrement `hintsUsed` (per P3.6).
- **FR-5.** After undo, `validationResult` MUST be recomputed.
- **FR-6.** Sequential undos MUST restore exact prior states in reverse order (LIFO).
- **FR-7.** Undo MUST be permitted even when the board was completed, to allow correcting an accidental final placement (board lock applies to new taps, not to undo).

## 6. Non-Functional Requirements

- **Performance** — O(1) pop + O(n) revalidation.
- **Accessibility** — Undo availability must be observable so the toolbar (P3.12) can enable/disable and VoiceOver can announce.
- **Maintainability** — Implement as the inverse of `apply(_:)` from P3.2; share a single `inverse(of:)` helper with redo.
- **Reliability** — State after undo MUST be byte-for-byte equal to the pre-action state (deterministic).

## 7. Acceptance Criteria

- **AC-1.** *Given* one `placeAnimal`, *when* `undo()`, *then* the cell is `.empty` and the action is on the redo stack.
- **AC-2.** *Given* an empty undo stack, *when* `undo()`, *then* nothing changes and no crash occurs.
- **AC-3.** *Given* a sequence of N actions, *when* `undo()` is called N times, *then* the board equals the initial empty board.
- **AC-4.** *Given* a `hint` action, *when* undone, *then* the cell is `.empty` and `hintsUsed` decreases by 1.
- **AC-5.** *Given* any undo, *when* it completes, *then* `validationResult` matches the restored board.

## 8. Data Model

```swift
extension GameSession {
    var canUndo: Bool { !undoStack.isEmpty }
    func undo() {
        guard let action = undoStack.popLast() else { return }   // FR-3
        applyInverse(of: action)                                 // FR-1, FR-4
        redoStack.append(action)                                 // FR-2
        revalidate()                                             // FR-5
    }
}
```

Reversal strategy: store enough context to invert each `GameAction`. For MVP the action + its target position is sufficient because state transitions are deterministic per cell (place↔empty, block↔empty, hint→empty + counter). If needed, capture the pre-mutation `CellState` alongside the action.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Inverse for `toggleBlocked` ambiguous | M | M | Reversal returns cell to pre-action state; capture prior state if not derivable |
| `hintsUsed` not decremented on undo | M | M | FR-4 + AC-4 test; centralize hint inverse |
| Undo blocked by post-win lock | L | M | FR-7 — lock guards new taps only |

## 1. Problem Statement

Individual UI components (`CellView`, `BoardView`, `GameToolbar`, `InputModeToggle`) and the engine (`GameSession`, `GameViewModel`) exist in isolation but are not composed into a playable screen. Without `GameView`, there is no end-to-end play loop вҖ” the **first playable milestone** defined in both the GDD and Implementation Roadmap cannot be reached.

## 2. Goals

- Compose `BoardView`, `InputModeToggle`, and `GameToolbar` into a single `GameView` screen.
- Bind all components to `GameViewModel` (P3.13) for live state and user intents.
- Deliver the v0.1 play loop: place/remove, mark X, validate, undo/redo/reset/hint, detect win.
- Fit the layout in portrait on iPhone SE through Pro Max without scrolling.
- Mark this story as the **PLAYABLE MILESTONE** вҖ” a human can complete one puzzle start-to-finish.

## 3. Non-Goals

- Win overlay UI (P3.15 вҖ” triggered by ViewModel but rendered separately).
- App entry point wiring (P3.16).
- Settings sheet (P4.4).
- Theme switcher (P4.2; v0.1 uses default Frogs theme).
- Placement/win animations (P5.1, P5.2).
- Sound effects (P4.5).

## 4. Personas & User Stories

- **As a player**, I want one screen with the board, mode toggle, and action buttons so I can play a full puzzle.
- **As a player**, I want immediate visual feedback when I violate a rule so I can adjust my strategy.
- **As a developer**, I want `GameView` to be the integration point that proves all Phase 3 engine + UI stories work together.
- **As a QA tester**, I want a deterministic play-to-win path on the bundled puzzle for UI automation (T-082).

## 5. Functional Requirements

- **FR-1.** `GameView` MUST compose, top to bottom: title/header, `InputModeToggle`, `BoardView`, `GameToolbar`.
- **FR-2.** `GameView` MUST own a `GameViewModel` (via `@State` or injection from P3.16).
- **FR-3.** `BoardView` MUST receive `cells`, `violatingPositions`, `selectedPosition`, and `onCellTap` from the ViewModel.
- **FR-4.** `InputModeToggle` MUST bind to `viewModel.inputMode`; disabled when `isCompleted`.
- **FR-5.** `GameToolbar` MUST wire `canUndo`, `canRedo`, `canHint`, and action callbacks to the ViewModel.
- **FR-6.** Cell taps MUST route through `viewModel.handleCellTap(at:)` respecting current `inputMode`.
- **FR-7.** When `viewModel.showWinScreen == true`, `GameView` MUST present `WinScreen` overlay (P3.15).
- **FR-8.** The board MUST include region colorblind patterns (P3.10) composed inside `BoardView`.
- **FR-9.** Layout MUST fit in portrait without vertical scroll on iPhone SE (375Г—667).
- **FR-10.** `GameView` MUST use the default bundled puzzle (`puzzle-001` or first loaded puzzle, P2.8).
- **FR-11.** Invalid placements MUST be allowed but highlighted (Formal Rules В§Placement Validation).
- **FR-12.** Accessibility identifiers MUST be set on board cells and toolbar buttons for UI tests (T-082).

## 6. Non-Functional Requirements

- **Performance** вҖ” 60fps during rapid tapping on iPhone SE.
- **Accessibility** вҖ” Full VoiceOver navigation: mode toggle вҶ’ board grid вҶ’ toolbar buttons.
- **Maintainability** вҖ” `UI/Screens/GameView.swift`; composition only, no engine logic.
- **Reliability** вҖ” No crash on any tap sequence including undo/redo chains and hint at limit.

## 7. Acceptance Criteria

- **AC-1 (PLAYABLE).** *Given* a fresh launch into `GameView`, *when* the player solves the bundled puzzle using place/remove/mark, *then* the win overlay appears (P3.15).
- **AC-2.** *Given* Place mode, *when* tapping an empty cell, *then* an animal appears on the board.
- **AC-3.** *Given* Mark mode, *when* tapping an empty cell, *then* an X appears.
- **AC-4.** *Given* two animals in the same row, *when* the board updates, *then* violating cells are highlighted.
- **AC-5.** *Given* a sequence of moves, *when* Undo is tapped, *then* the board reverts correctly.
- **AC-6.** *Given* a sequence of undos, *when* Redo is tapped, *then* moves are reapplied.
- **AC-7.** *Given* a partially filled board, *when* Reset is tapped, *then* all cells are empty and counters cleared (P3.5).
- **AC-8.** *Given* `hintsUsed < 3`, *when* Hint is tapped, *then* one solution cell is revealed (P3.6).
- **AC-9.** *Given* iPhone SE in portrait, *when* `GameView` renders, *then* the full layout is visible without scrolling.
- **AC-10.** *Given* a completed puzzle, *when* the player taps cells, *then* no changes occur (board locked).

## 8. Data Model

No new types. Composition wiring:

```swift
// UI/Screens/GameView.swift
struct GameView: View {
    @State private var viewModel: GameViewModel

    init(puzzle: Puzzle) {
        _viewModel = State(initialValue: GameViewModel(puzzle: puzzle))
    }

    var body: some View {
        VStack(spacing: AppSpacing.md) {
            Text("Animal Doku")
                .font(AppTypography.title)

            InputModeToggle(
                mode: $viewModel.inputMode,
                isEnabled: !viewModel.isCompleted
            )

            BoardView(
                puzzle: viewModel.puzzle,
                cells: viewModel.cells,
                selectedPosition: viewModel.selectedPosition,
                violatingPositions: viewModel.violatingPositions,
                animalIcon: Image("frog"),  // default theme placeholder
                onCellTap: { viewModel.handleCellTap(at: $0) }
            )
            .accessibilityIdentifier("gameBoard")

            GameToolbar(
                canUndo: viewModel.canUndo,
                canRedo: viewModel.canRedo,
                canReset: !viewModel.isCompleted,
                canHint: viewModel.canHint,
                hintsRemaining: viewModel.hintsRemaining,
                onUndo: { viewModel.undo() },
                onRedo: { viewModel.redo() },
                onReset: { viewModel.reset() },
                onHint: { viewModel.requestHint() }
            )
        }
        .padding(AppSpacing.md)
        .sheet(isPresented: $viewModel.showWinScreen) {
            WinScreen(onPlayAgain: { viewModel.reset() })
        }
    }
}
```

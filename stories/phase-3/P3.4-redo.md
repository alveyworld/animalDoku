## 1. Problem Statement

After undoing one or more moves, players may want to reapply them without re-tapping. `GameSession` produces a redo stack during undo (P3.3) but cannot yet replay it. Redo completes the standard undo/redo pair expected by users.

## 2. Goals

- Reapply the most recently undone action, restoring the state that existed before the undo.
- Keep the redo stack consistent: cleared whenever a new action is taken after undo (already enforced in P3.2).
- Make redo a safe no-op when the redo stack is empty.

## 3. Non-Goals

- Recording new actions (P3.2) or reversing them (P3.3).
- Toolbar Redo button UI (P3.12).

## 4. Personas & User Stories

- **As a player**, I want to redo a move I just undid so I can compare board states quickly.
- **As a player**, I expect redo to be discarded once I make a new move, matching standard editor behavior.
- **As a developer**, I want redo to reuse the same `apply(_:)` path as normal actions for consistency.

## 5. Functional Requirements

- **FR-1.** `redo()` MUST pop the last action from `redoStack` and reapply it (same effect as originally performing it).
- **FR-2.** The reapplied action MUST be pushed back onto `undoStack`.
- **FR-3.** `redo()` on an empty `redoStack` MUST be a no-op.
- **FR-4.** The `redoStack` MUST be cleared when a new action is recorded after an undo (enforced in P3.2 FR-9; verified here).
- **FR-5.** Redo MUST correctly reapply every action type, including `hint` (re-increment `hintsUsed`).
- **FR-6.** After redo, `validationResult` MUST be recomputed.
- **FR-7.** Redo MUST be rejected when the board is currently locked by completion (a completed state has no pending redo path that would un-complete then re-complete unexpectedly); if redo would re-complete the board, the win flow (P3.15) MUST trigger consistently.

## 6. Non-Functional Requirements

- **Performance** — O(1) pop + O(n) revalidation.
- **Accessibility** — `canRedo` observable for toolbar enable/disable (P3.12).
- **Maintainability** — Reuse `apply(_:)` from P3.2; mirror `undo()` structure.
- **Reliability** — Redo then undo MUST be a round-trip returning to the same state.

## 7. Acceptance Criteria

- **AC-1.** *Given* an undone `placeAnimal`, *when* `redo()`, *then* the cell is `.animal` again and the action returns to the undo stack.
- **AC-2.** *Given* an empty redo stack, *when* `redo()`, *then* nothing changes and no crash occurs.
- **AC-3.** *Given* an undo followed by a new placement, *when* the redo stack is inspected, *then* it is empty (AC ties to P3.2 AC-8).
- **AC-4.** *Given* an undone `hint`, *when* redone, *then* `hintsUsed` increments back and the cell is `.animal`.
- **AC-5.** *Given* undo then redo of the same action, *when* both complete, *then* the board equals the pre-undo state.

## 8. Data Model

```swift
extension GameSession {
    var canRedo: Bool { !redoStack.isEmpty }
    func redo() {
        guard let action = redoStack.popLast() else { return }  // FR-3
        apply(action)                                           // FR-1, FR-5 (reuses P3.2)
        // apply(_:) already appends to undoStack and revalidates (FR-2, FR-6)
    }
}
```

Note: because `apply(_:)` normally clears `redoStack`, `redo()` must pop first (or use a dedicated internal apply that preserves the stack) to avoid wiping pending redo entries.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `apply(_:)` wipes redo stack during redo | M | H | Pop redo entry before applying, or use internal apply that does not clear redo |
| Redo re-completes board without win flow | L | M | Route completion detection through one path (P3.15) regardless of trigger |
| Hint counter drift on redo | M | M | FR-5 + AC-4 test |

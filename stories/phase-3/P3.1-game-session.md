## 1. Problem Statement

There is no runtime container that holds the live board, move history, hint count, timer, and completion state for an active puzzle. Without `GameSession`, no player action (place, remove, mark, undo, redo, reset, hint) can be recorded or reasoned about, and the UI has no source of truth to bind to. This is the engine-layer foundation for all of Phase 3.

## 2. Goals

- Create a pure-Swift `GameSession` that initializes from a `Puzzle` with an all-empty board.
- Hold current cells, undo/redo stacks, `hintsUsed`, `elapsedSeconds`, input mode, and completion flag.
- Expose the latest `ValidationResult` so callers can render violations and detect completion.
- Keep the type free of SwiftUI imports so it is fully unit-testable (P3.7).

## 3. Non-Goals

- Implementing the mutation actions themselves (P3.2), undo/redo (P3.3/P3.4), reset (P3.5), or hint (P3.6) — this story only establishes the container, initialization, and shared state.
- Timer ticking/background-pause wiring (tracked in session field here; service ships in P4.6).
- Persistence / `SaveGame` round-trip (v1.1, P4.7).
- Any UI binding (`GameViewModel` is P3.13).

## 4. Personas & User Stories

- **As a developer**, I want a single state container per puzzle attempt so that all actions mutate one consistent source of truth.
- **As a developer**, I want the session to start from an empty board per Formal Rules so that MVP puzzles have no pre-filled givens.
- **As a player** (indirect), I want my board, hints, and elapsed time tracked accurately for the whole attempt.

## 5. Functional Requirements

Written in MUST / SHOULD / MAY (RFC 2119) form.

- **FR-1.** `GameSession` MUST initialize from a `Puzzle`, building a `size × size` grid of `Cell` values with `state == .empty` and the correct `regionId` per the puzzle's regions.
- **FR-2.** `GameSession` MUST expose current board state as an ordered, indexable collection of `Cell` (e.g. `cells: [Cell]` in row-major order, plus a lookup by `Position`).
- **FR-3.** `GameSession` MUST hold `undoStack` and `redoStack` of `GameAction` (mutated by P3.3/P3.4).
- **FR-4.** `GameSession` MUST hold `hintsUsed: Int` (default 0) and `elapsedSeconds: Int` (default 0).
- **FR-5.** `GameSession` MUST hold the current `inputMode` (Place default, Mark) per Formal Rules Interaction Model.
- **FR-6.** `GameSession` MUST expose a `completed: Bool` (or equivalent) derived from the latest validation.
- **FR-7.** `GameSession` MUST recompute and expose a `ValidationResult` whenever board state changes (delegating to the `Validator` from P2.4–P2.7).
- **FR-8.** `GameSession` MUST retain the original `Puzzle` immutably for reset (P3.5) and hint (P3.6) reference.
- **FR-9.** The type MUST NOT import SwiftUI/UIKit.

## 6. Non-Functional Requirements

- **Performance** — Validation triggered on state change is O(n) per Formal Rules (n = size); trivial at 8×8.
- **Security** — N/A (local, offline).
- **Privacy** — No PII; session is in-memory only for v0.1.
- **Accessibility** — N/A at this layer (surfaced by UI stories).
- **Maintainability** — Engine layer only; no view concerns. Follows `Engine/` module boundary (P1.2).
- **Reliability** — Initialization MUST be deterministic for a given puzzle (same cells, same order).
- **Testability** — All state transitions observable via public read-only properties for P3.7.

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid `Puzzle` of size 8, *when* a `GameSession` is created, *then* it has 64 cells, all `.empty`, each with the `regionId` matching the puzzle regions.
- **AC-2.** *Given* a fresh session, *when* inspected, *then* `hintsUsed == 0`, `elapsedSeconds == 0`, `undoStack`/`redoStack` are empty, `inputMode == .place`, and `completed == false`.
- **AC-3.** *Given* a session, *when* the board changes, *then* `validationResult` reflects the current board (`isValid`, `isComplete`, `violations`).
- **AC-4.** *Given* a session, *when* the original puzzle is read, *then* it is unchanged regardless of board mutations.

## 8. Data Model

Uses types from P2.1/P2.2/P2.7. New container type:

```swift
// Engine/GameSession.swift
enum InputMode { case place, mark }

final class GameSession {
    let puzzle: Puzzle                 // immutable reference (FR-8)
    private(set) var cells: [Cell]     // row-major, size*size
    private(set) var undoStack: [GameAction] = []
    private(set) var redoStack: [GameAction] = []
    private(set) var hintsUsed: Int = 0
    private(set) var elapsedSeconds: Int = 0
    var inputMode: InputMode = .place
    private(set) var validationResult: ValidationResult
    var completed: Bool { validationResult.isComplete }

    init(puzzle: Puzzle, validator: Validating = Validator()) { /* build empty board */ }
    func cell(at position: Position) -> Cell { /* lookup */ }
    // Mutations (place/remove/block, undo, redo, reset, hint) added in P3.2–P3.6.
}
```

Note: `class` (reference type) chosen so the `GameViewModel` (P3.13) can wrap a single shared instance; hint limit constant (`maxHints = 3`) defined here or in `HintService` (P3.6).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Region/cell mapping mismatch at init | M | H | Assert regions partition all cells (invariant from Formal Rules); unit test in P3.7 |
| Value vs. reference semantics confusion | M | M | Use `class` for the session; keep `Cell`/`Puzzle` as value types |
| Validation not re-run after a mutation | M | M | Centralize validation in one private `revalidate()` called by every mutation (P3.2+) |

## Note on tap intent

The public entry point is a single **tap-at-position** intent; the resulting mutation depends on `inputMode` and the tapped cell's state, per the Formal Rules Interaction Model table. This story implements that mapping plus the low-level mutations.

---

## 1. Problem Statement

`GameSession` (P3.1) holds board state but cannot yet change it. Players need to place and remove animals in Place mode and toggle/clear blocked marks in Mark mode, with each change validated and recorded in history for undo/redo. Without these actions, the game is not interactive.

## 2. Goals

- Implement the two-mode tap behavior exactly as defined in Formal Rules § Interaction Model.
- Mutate cell state through `GameSession`, recording each change as a `GameAction` on the undo stack.
- Re-run validation after every mutation and expose the new `ValidationResult`.
- Enforce placement rules (blocked cells reject placement; board locks after completion).

## 3. Non-Goals

- Undo (P3.3), redo (P3.4), reset (P3.5), hint (P3.6) — those consume the history this story records.
- The toolbar/mode-toggle UI (P3.11, P3.12).
- Auto-clearing a blocked mark when placing (explicitly disallowed by Formal Rules).

## 4. Personas & User Stories

- **As a player in Place mode**, I want to tap an empty cell to place an animal and tap it again to remove it.
- **As a player in Mark mode**, I want to tap a cell to mark it impossible (X) and tap again to clear it, so I can reason about the puzzle.
- **As a player**, I want invalid placements allowed but highlighted so I can learn from mistakes without being blocked.
- **As a developer**, I want every mutation recorded as a reversible action so undo/redo work uniformly.

## 5. Functional Requirements

Behavior MUST match this table (Formal Rules § Input Modes):

| Mode | Tap empty | Tap blocked (X) | Tap animal |
|------|-----------|-----------------|------------|
| **Place** (default) | Place animal | No effect | Remove animal |
| **Mark** | Toggle blocked (X) | Clear X → empty | No effect |

- **FR-1.** In **Place** mode, tapping an `.empty` cell MUST set it to `.animal` and push a `placeAnimal(position)` action.
- **FR-2.** In **Place** mode, tapping an `.animal` cell MUST set it to `.empty` and push a `removeAnimal(position)` action.
- **FR-3.** In **Place** mode, tapping a `.blocked` cell MUST have **no effect** (no state change, no history entry).
- **FR-4.** In **Mark** mode, tapping an `.empty` cell MUST set it to `.blocked` and push a `toggleBlocked(position)` action.
- **FR-5.** In **Mark** mode, tapping a `.blocked` cell MUST set it to `.empty` and push the reversing `toggleBlocked(position)` action.
- **FR-6.** In **Mark** mode, tapping an `.animal` cell MUST have **no effect**.
- **FR-7.** Placing an animal MUST NOT auto-clear an existing blocked mark; blocked cells reject placement (consequence of FR-3).
- **FR-8.** Invalid placements MUST be **allowed** and recorded; violations are surfaced via `ValidationResult`, not prevented.
- **FR-9.** Any successful mutation MUST push onto `undoStack` and MUST clear `redoStack`.
- **FR-10.** After the board is complete (`validationResult.isComplete == true`), all mutating taps MUST be rejected (board locked) until Reset/Play Again (P3.5, P3.15).
- **FR-11.** Every successful mutation MUST trigger revalidation so `validationResult` is current.

## 6. Non-Functional Requirements

- **Performance** — Each action is O(1) mutation + O(n) revalidation; imperceptible at 8×8.
- **Security** — N/A.
- **Privacy** — N/A.
- **Accessibility** — State changes must be observable so VoiceOver labels update (surfaced in P3.8/P3.14).
- **Maintainability** — Single `tap(at:)` intent method routing on `inputMode`; low-level `apply(_ action:)` shared with undo/redo.
- **Reliability** — Mutations are deterministic and idempotent per the table (no-effect cases truly no-op).

## 7. Acceptance Criteria

- **AC-1.** *Given* Place mode and an empty cell, *when* tapped, *then* the cell becomes `.animal` and one `placeAnimal` action is on the undo stack.
- **AC-2.** *Given* Place mode and an animal cell, *when* tapped, *then* the cell becomes `.empty` and a `removeAnimal` action is recorded.
- **AC-3.** *Given* Place mode and a blocked cell, *when* tapped, *then* nothing changes and no action is recorded.
- **AC-4.** *Given* Mark mode and an empty cell, *when* tapped, *then* it becomes `.blocked`; *when* tapped again, *then* it returns to `.empty`.
- **AC-5.** *Given* Mark mode and an animal cell, *when* tapped, *then* nothing changes.
- **AC-6.** *Given* two animals placed in the same row, *when* validated, *then* `isValid == false` with a `row` violation listing both positions (per AC-V.1).
- **AC-7.** *Given* a completed board, *when* any cell is tapped, *then* no mutation occurs (board locked).
- **AC-8.** *Given* an action after an undo, *when* a new placement is made, *then* the redo stack is cleared.

## 8. Data Model

Consumes `GameAction` (P2.2) and mutates `GameSession.cells` (P3.1):

```swift
enum GameAction {                 // from P2.2
    case placeAnimal(Position)
    case removeAnimal(Position)
    case toggleBlocked(Position)
    case hint(Position)           // used by P3.6
}

extension GameSession {
    func tap(at position: Position) { /* route on inputMode + current state */ }
    func apply(_ action: GameAction) { /* low-level mutate + revalidate; shared w/ undo/redo */ }
}
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Mode/state routing bugs (16-cell matrix) | M | H | Table-driven tests covering every mode × state cell (P3.7) |
| Blocked-cell placement leak | L | M | Explicit FR-3/FR-7 test; no auto-clear |
| Redo stack not cleared on new action | M | M | Centralize in `apply(_:)`; AC-8 test |
| Mutations allowed after win | M | M | Guard on `completed` at top of `tap(at:)` (FR-10) |

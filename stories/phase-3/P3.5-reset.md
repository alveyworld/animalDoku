## 1. Problem Statement

Players need a fast way to start over without relaunching the app. After hints, mistakes, or completion, the board must return to its initial empty state and all session counters/history must be cleared. Formal Rules specifies reset clears undo/redo stacks, hint count, and timer — but not theme or settings.

## 2. Goals

- Restore the board to the initial all-empty state (same as session init from P3.1).
- Clear undo/redo stacks, reset `hintsUsed` to 0, reset `elapsedSeconds` to 0.
- Clear completion state so the board is editable again.
- Preserve the original `Puzzle`, `inputMode`, and app-level settings (theme, sound).

## 3. Non-Goals

- Reloading a different puzzle (v1.1, P4.8).
- Resetting theme or settings (out of scope per Formal Rules).
- Confirmation dialog UX (optional polish; v0.1 may reset immediately on tap).

## 4. Personas & User Stories

- **As a player**, I want to reset the puzzle so I can try again from scratch.
- **As a player on the win screen**, I want Play Again to reset the current puzzle (P3.15 delegates to this).
- **As a developer**, I want reset to be a single session method reused by toolbar and win overlay.

## 5. Functional Requirements

- **FR-1.** `reset()` MUST set every cell to `.empty` (initial state for MVP; `initialPlacements` is always `[]`).
- **FR-2.** `reset()` MUST clear `undoStack` and `redoStack`.
- **FR-3.** `reset()` MUST set `hintsUsed` to 0.
- **FR-4.** `reset()` MUST set `elapsedSeconds` to 0.
- **FR-5.** `reset()` MUST recompute `validationResult` (expect `isComplete == false` on empty board).
- **FR-6.** `reset()` MUST NOT change the bound `Puzzle`, `inputMode`, or any `SettingsStore` values.
- **FR-7.** After reset, the board MUST be unlocked for new taps (completion lock lifted).
- **FR-8.** `reset()` SHOULD be callable at any time, including while the win overlay is shown.

## 6. Non-Functional Requirements

- **Performance** — O(n) to clear cells + O(n) revalidation; trivial at 8×8.
- **Accessibility** — Reset availability always true; toolbar (P3.12) and win Play Again (P3.15) expose it.
- **Maintainability** — Reuse the same initialization path as `init(puzzle:)` to avoid drift.
- **Reliability** — Repeated resets are idempotent.

## 7. Acceptance Criteria

- **AC-1.** *Given* a partially filled board with history and `hintsUsed > 0`, *when* `reset()`, *then* all cells are `.empty`, stacks are empty, `hintsUsed == 0`, `elapsedSeconds == 0`.
- **AC-2.** *Given* a completed board, *when* `reset()`, *then* `completed == false` and taps are accepted again.
- **AC-3.** *Given* a reset, *when* the puzzle reference is inspected, *then* it is unchanged.
- **AC-4.** *Given* a reset, *when* `inputMode` is inspected, *then* it retains its pre-reset value (not forced back to Place).
- **AC-5.** *Given* two consecutive `reset()` calls, *when* inspected, *then* the board state is identical (idempotent).

## 8. Data Model

```swift
extension GameSession {
    func reset() {
        cells = Self.makeEmptyCells(from: puzzle)  // FR-1; shared with init
        undoStack.removeAll()                      // FR-2
        redoStack.removeAll()                      // FR-2
        hintsUsed = 0                              // FR-3
        elapsedSeconds = 0                         // FR-4
        revalidate()                               // FR-5, FR-7
    }
}
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Reset leaves stale completion/win overlay | M | M | ViewModel observes `completed`; overlay dismisses when `completed` becomes false |
| Init vs reset cell-building drift | M | M | Single `makeEmptyCells(from:)` helper (FR-1) |
| Timer not zeroed | L | M | FR-4 + AC-1 test |

## 1. Problem Statement

The `GameSession` engine encodes all player interactions, history, hints, and completion logic. Without automated tests, regressions in place/remove/mark, undo/redo, reset, and hint behavior will reach the UI undetected. This story delivers the test suite that proves the engine is correct before wiring the playable milestone (P3.14).

## 2. Goals

- Create `Tests/GameSessionTests.swift` with comprehensive coverage of P3.1–P3.6.
- Use a small, deterministic test puzzle fixture (inline or bundled test JSON).
- Achieve high branch coverage on session mutation paths.
- Mirror Formal Rules acceptance criteria (AC-H, interaction table, reset scope).

## 3. Non-Goals

- Validator tests (P2.9).
- UI tests (P6.2).
- Puzzle integrity / single-solution tests (P6.1).
- Performance/load testing.

## 4. Personas & User Stories

- **As a developer**, I want GameSession tests so I can refactor safely.
- **As a developer**, I want tests aligned to Formal Rules AC so spec drift is caught in CI.
- **As a player** (indirect), I want reliable undo/hint/reset behavior.

## 5. Functional Requirements

- **FR-1.** Tests MUST run in the unit test target configured in P1.3.
- **FR-2.** Tests MUST cover session initialization (P3.1 AC-1 through AC-4).
- **FR-3.** Tests MUST cover the full Place/Mark interaction matrix (P3.2 AC-1 through AC-7).
- **FR-4.** Tests MUST cover undo sequences including empty-stack no-op and hint undo (P3.3 AC-1 through AC-5).
- **FR-5.** Tests MUST cover redo including stack-clear-on-new-action (P3.4 AC-1 through AC-5).
- **FR-6.** Tests MUST cover reset clearing hints/timer/history (P3.5 AC-1 through AC-5).
- **FR-7.** Tests MUST cover hint limits, blocked rejection, and row-major selection (P3.6 AC-1 through AC-7).
- **FR-8.** Tests SHOULD use a dedicated `TestPuzzleFactory` or inline fixture, not the production MVP puzzle, to keep cases minimal and deterministic.

## 6. Non-Functional Requirements

- **Performance** — Full suite completes in < 5 seconds on CI hardware.
- **Maintainability** — Group tests by story (`// MARK: - P3.2 Place/Remove/Block`); shared helpers for board assertions.
- **Reliability** — No flaky timing tests (timer tick tests deferred to P4.6).

## 7. Acceptance Criteria

- **AC-1.** *Given* the test target, *when* `xcodebuild test` runs, *then* all `GameSessionTests` pass.
- **AC-2.** *Given* the suite, *when* coverage is measured, *then* `GameSession.swift` and `HintService.swift` exceed 90% line coverage (target; 80% minimum).
- **AC-3.** *Given* each Formal Rules hint AC (AC-H.1–H.3), *when* mapped, *then* at least one automated test exists.
- **AC-4.** *Given* a scripted solve of the test puzzle via placements only, *when* the solution is reached, *then* `completed == true`.

## 16. Test Plan

This story **is** the test plan for the engine layer. Organized suites:

- **Unit — Initialization** — Empty board, region IDs, zeroed counters, initial validation.
- **Unit — Place/Remove/Block** — Mode × state matrix; violation after conflict; post-win lock.
- **Unit — Undo** — Per-action reversal; N-step walk-back; hint counter; empty stack.
- **Unit — Redo** — Reapply; round-trip; clear on new action; hint re-increment.
- **Unit — Reset** — Clears board, stacks, hints, timer; preserves puzzle/mode; idempotent.
- **Unit — Hint** — AC-H.1–3; selection; row-major; max 3; undo; completion trigger.
- **Unit — Solve path** — Place all solution cells → `completed == true`.
- **Manual exploratory** — N/A.

## 1. Problem Statement

Apple requires a privacy manifest (`PrivacyInfo.xcprivacy`) and App Store privacy answers for new app submissions, including declarations for any "required-reason" APIs. Animal Doku is a fully offline, no-account game that collects no personal data, but Apple still expects an accurate manifest and a truthful "Data Not Collected" declaration. Without a correct manifest, TestFlight upload and App Store submission are blocked or rejected. This story produces the manifest and privacy answers that reflect the app's zero-collection reality.

## 2. Goals

- Add a `PrivacyInfo.xcprivacy` manifest to the app target declaring no tracking and no collected data types.
- Declare any required-reason API usage (e.g. `UserDefaults`) with the correct approved reason code, or confirm none are used.
- Prepare accurate App Store Connect "App Privacy" answers ("Data Not Collected").
- Verify no bundled third-party SDKs introduce undeclared data collection or tracking.
- Ensure the manifest passes Xcode archive validation with no privacy warnings.

## 3. Non-Goals

- A user-facing privacy policy web page (may be required at submission — tracked in P7.3).
- Analytics, crash reporting, or any SDK that would change the privacy posture (out of scope for MVP/v1.0).
- ATT (App Tracking Transparency) prompt — not applicable; the app does no tracking.
- Cloud save / accounts (future features that would revisit this manifest).

## 4. Personas & User Stories

- **As a player**, I want assurance the app collects no personal data so that I can play privately.
- **As the developer**, I want a correct privacy manifest so that submission is not rejected.
- **As an App Store reviewer**, I want accurate privacy declarations so that the listing matches app behavior.

## 5. Functional Requirements

- **FR-1.** The app target MUST include `PrivacyInfo.xcprivacy` in the bundle.
- **FR-2.** `NSPrivacyTracking` MUST be `false` and `NSPrivacyTrackingDomains` MUST be empty (no tracking).
- **FR-3.** `NSPrivacyCollectedDataTypes` MUST be empty (or accurately reflect any data collected — none for v1.0).
- **FR-4.** `NSPrivacyAccessedAPITypes` MUST list every required-reason API used with an approved reason code; if `UserDefaults` is used (P4.3), it MUST be declared with reason `CA92.1` (access info stored solely by this app).
- **FR-5.** The manifest MUST be valid plist and pass Xcode archive validation with no privacy-related warnings.
- **FR-6.** App Store Connect "App Privacy" answers MUST be prepared as "Data Not Collected" consistent with the manifest.
- **FR-7.** Any third-party binary/SPM dependency MUST be reviewed; if none collect data, document "no third-party SDKs" (v1.0 uses none).

## 6. Non-Functional Requirements

- **Performance** — No runtime cost; manifest is static metadata.
- **Security** — No secrets in the manifest; it is public metadata.
- **Privacy & Compliance** — Aligns with Apple privacy requirements; app collects no PII (COPPA/GDPR posture: no data collected, suitable for all ages).
- **Accessibility** — N/A.
- **Maintainability** — Manifest lives at a known path; update if future features (cloud save, analytics) change data usage.
- **Backward compatibility** — Manifest is additive; safe on iOS 17+.

## 7. Acceptance Criteria

- **AC-1.** *Given* the app bundle, *when* inspected, *then* `PrivacyInfo.xcprivacy` is present and is valid plist.
- **AC-2.** *Given* the manifest, *when* read, *then* `NSPrivacyTracking` is `false` and no tracking domains are listed.
- **AC-3.** *Given* the manifest, *when* read, *then* `NSPrivacyCollectedDataTypes` is empty (matching actual behavior).
- **AC-4.** *Given* `UserDefaults` usage, *when* the manifest is checked, *then* `NSPrivacyAccessedAPITypes` declares it with reason `CA92.1` (or the required-reason list is empty if unused).
- **AC-5.** *Given* an Xcode archive, *when* validated, *then* no privacy manifest warnings are produced.
- **AC-6.** *Given* App Store Connect, *when* privacy answers are entered, *then* they state "Data Not Collected" consistent with the manifest.

## 8. Data Model

No production types. Manifest content (illustrative):

```xml
<!-- PrivacyInfo.xcprivacy -->
<dict>
  <key>NSPrivacyTracking</key><false/>
  <key>NSPrivacyTrackingDomains</key><array/>
  <key>NSPrivacyCollectedDataTypes</key><array/>
  <key>NSPrivacyAccessedAPITypes</key>
  <array>
    <dict>
      <key>NSPrivacyAccessedAPIType</key>
      <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
      <key>NSPrivacyAccessedAPITypeReasons</key>
      <array><string>CA92.1</string></array>
    </dict>
  </array>
</dict>
```

- Shared infra needed: App Store Connect access (for privacy answers).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Missing required-reason API declaration | M | H | Audit API usage (UserDefaults, file timestamps); declare with correct reason codes |
| Future SDK adds hidden collection | L | H | Review dependencies at each release; keep v1.0 SDK-free |
| Privacy answers mismatch manifest | M | M | Derive answers directly from manifest; single source of truth |
| Apple updates required-reason list | L | M | Recheck Apple docs at submission time |

## 15. Rollout Plan

- No feature flag. Include manifest in the v1.0 archive.
- Rollback: manifest is metadata; correct and re-archive if validation flags an issue.
- Comms: none needed; privacy label is public via App Store.

## 16. Test Plan

- **Validation** — Archive in Xcode Organizer; confirm no privacy warnings (AC-5).
- **Inspection** — Verify plist keys/values match required declarations (AC-1–AC-4).
- **Dependency review** — Confirm no third-party SDKs collect data (FR-7).
- **Manual** — Enter and screenshot App Store Connect privacy answers (AC-6).

## 17. Documentation & Training

- Document the manifest location and how to update it when adding features that touch data (cloud save, analytics).
- Note the required-reason API mapping for future contributors.

## 18. Open Questions

1. Is a hosted privacy policy URL required at submission for an all-ages, no-data app? **Decided: prepare in P7.3** (short static policy recommended at submission).
2. Confirm whether any file-timestamp or system-boot APIs are transitively used? **Audited:** `UserDefaults` (SettingsStore) + `ProcessInfo.systemUptime` (TimerService). No File Timestamp / Disk Space / Active Keyboard APIs. No third-party SDKs.

## 1. Problem Statement

Before a public App Store release, the app should be validated on real devices by real testers to catch device-specific bugs, onboarding confusion, and crashes that simulators and unit tests miss. TestFlight is Apple's beta distribution channel, but it requires a signed, uploaded build with a complete icon, privacy manifest, and App Store Connect setup. This story establishes the TestFlight pipeline and ships a beta to internal and external testers, closing the loop between "builds on my machine" and "works for players."

## 2. Goals

- Create the App Store Connect app record and configure signing/provisioning for distribution.
- Archive, sign, and upload a build to TestFlight.
- Distribute to an internal tester group and an external beta group (with beta review).
- Provide clear test instructions and a feedback channel; triage incoming feedback and crash reports.
- Confirm the build passes TestFlight processing and beta app review.

## 3. Non-Goals

- Public App Store release and full store listing (P7.3).
- Building app features — this is distribution/QA, not development.
- Automated CI/CD upload pipeline (nice-to-have; may be manual `xcodebuild`/Xcode Organizer for v1.1).
- Marketing screenshots and localized metadata (P7.3).

## 4. Personas & User Stories

- **As a beta tester**, I want to install the app easily and report issues so that bugs are fixed before launch.
- **As the developer**, I want real-device crash reports and feedback so that I can stabilize before release.
- **As a product owner**, I want a beta gate so that we ship v1.0 with fewer surprises.

## 5. Functional Requirements

- **FR-1.** An App Store Connect app record MUST exist with the final bundle identifier and app name.
- **FR-2.** A distribution signing certificate and provisioning profile MUST be configured (automatic signing acceptable).
- **FR-3.** A release-configuration archive MUST be produced, incorporating the app icon (P5.4) and privacy manifest (P7.1).
- **FR-4.** The build MUST upload to TestFlight and complete processing without errors.
- **FR-5.** The build MUST be distributed to at least one internal group and one external group; external distribution MUST pass TestFlight beta review.
- **FR-6.** Test instructions and "What to Test" notes MUST be provided; a feedback mechanism (TestFlight feedback/screenshots) MUST be enabled.
- **FR-7.** Crash reports and feedback MUST be collected and triaged; blocking issues MUST be filed against the relevant stories before P7.3.
- **FR-8.** Build/version numbers MUST follow a consistent scheme (e.g. `CFBundleShortVersionString` 1.0, incrementing `CFBundleVersion`).

## 6. Non-Functional Requirements

- **Performance** — Beta build behaves like release config (optimizations on); verify launch time and smoothness on device.
- **Security** — Signing assets stored securely; no debug entitlements or test backdoors in the archived build (strip the `-uiTestPuzzle` affordance or gate it to DEBUG).
- **Privacy & Compliance** — Privacy manifest (P7.1) present; beta export-compliance question answered (uses only exempt/no encryption).
- **Accessibility** — Ship the accessibility-audited build (P6.3) so testers exercise a11y too.
- **Reliability** — Build must be crash-free on cold launch across the beta device pool.
- **Observability** — Enable TestFlight crash collection; monitor feedback.

## 7. Acceptance Criteria

- **AC-1.** *Given* an archived release build, *when* uploaded, *then* it appears in App Store Connect and completes TestFlight processing without errors.
- **AC-2.** *Given* the processed build, *when* assigned to an internal group, *then* internal testers can install and launch it on device.
- **AC-3.** *Given* external distribution, *when* submitted for beta review, *then* the build is approved and installable by external testers.
- **AC-4.** *Given* the beta, *when* testers play, *then* crash reports and feedback are received in App Store Connect.
- **AC-5.** *Given* collected feedback, *when* triaged, *then* blocking issues are filed/tracked before proceeding to P7.3.
- **AC-6.** *Given* the archived build, *when* inspected, *then* it contains the final icon (P5.4) and privacy manifest (P7.1) and no DEBUG-only test hooks.

N/A. Distribution tooling only:

```bash

```

## 12. Integration Points

- `AnimalDoku.xcodeproj` signing settings (P1.1)

## 13. Dependencies & Sequencing

- Must ship after: P5.4 (icon/launch), P7.1 (privacy manifest); ideally after P6.2/P6.3/P6.4 QA passes.
- Must ship before: P7.3 (public submission) — beta validates the release candidate.
- Shared infra needed: Apple Developer Program membership, App Store Connect access, real test devices.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Signing/provisioning failures | M | H | Use automatic signing; verify certificates before archiving |
| Beta review rejection | M | M | Ensure icon, privacy, and functionality complete; respond promptly |
| Debug test hooks ship in beta | M | H | Gate `-uiTestPuzzle` and identifiers behind DEBUG; verify in archive (AC-6) |
| Low tester engagement | M | L | Provide clear instructions and a short "What to Test" checklist |
| Export-compliance misanswer | L | M | Confirm app uses only exempt encryption (HTTPS/none); set Info.plist key |

## 15. Rollout Plan

- Distribute internal first; fix blockers; then expand to external group.
- Version scheme: marketing 1.0, incrementing build numbers per upload.
- Rollback: expire a bad build in TestFlight; upload a fixed build.
- GA gate: no crash-on-launch, no P0/P1 open, before moving to P7.3.

## 16. Test Plan

- **Distribution** — Upload succeeds and processes (AC-1); internal install (AC-2); external beta review pass (AC-3).
- **Device QA** — Testers run the core loop on varied devices (ties to P6.4 matrix).
- **Feedback loop** — Confirm crash/feedback capture (AC-4) and triage (AC-5).
- **Build hygiene** — Inspect archive for icon, manifest, no DEBUG hooks (AC-6).
- **Manual exploratory** — Structured "What to Test" checklist for testers.

## 17. Documentation & Training

- Document the archive → upload → distribute runbook.
- Provide a tester onboarding note (install TestFlight, redeem invite, how to send feedback).

## 18. Open Questions

1. Internal-only vs. external beta for v1.1 — recommend both, starting internal.
2. Manual upload vs. automated CI/CD (Xcode Cloud/Fastlane) — recommend manual for first beta, automate later.

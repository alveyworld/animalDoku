## 1. Problem Statement

The GDD requires a high-contrast mode with text/icon contrast ≥ 4.5:1 and increased border weight for improved visibility. Design tokens define high-contrast variants (P1.4) and `SettingsStore` persists the toggle (P4.3), but views still render default colors. This story wires the toggle through the UI so board, cells, toolbar, and settings meet the accessibility target.

## 2. Goals

- Apply high-contrast design tokens app-wide when `SettingsStore.highContrastEnabled` is true.
- Achieve ≥ 4.5:1 contrast for text and icons against their backgrounds per GDD.
- Increase border weight on cells and regions for clearer separation.
- Preserve colorblind region patterns (P3.10) in high-contrast mode.

## 3. Non-Goals

- System-wide iOS Increase Contrast setting integration beyond reading our toggle (may complement later).
- Dark mode as a separate theme.
- Changing puzzle logic or region colors in JSON data.

## 4. Personas & User Stories

- **As a player with low vision**, I want stronger contrast so that I can read the board and UI.
- **As a player**, I want to toggle high contrast in Settings and see the change immediately.
- **As a developer**, I want views to read semantic tokens so contrast mode is one switch, not per-view hacks.

## 5. Functional Requirements

- **FR-1.** When `highContrastEnabled` is true, the app MUST use high-contrast token variants from P1.4 for backgrounds, borders, text, and icons.
- **FR-2.** Text and icon contrast against adjacent backgrounds MUST be ≥ 4.5:1 (WCAG AA for normal text/icons).
- **FR-3.** Cell and region borders MUST use increased weight in high-contrast mode.
- **FR-4.** Region patterns (P3.10) MUST remain visible and distinguishable in high-contrast mode.
- **FR-5.** Toggling high contrast in Settings MUST apply immediately without relaunch.
- **FR-6.** Violation highlighting MUST remain clearly visible in high-contrast mode.

## 6. Non-Functional Requirements

- **Performance** — Token swap is a re-render only; no measurable frame drop.
- **Accessibility** — Meets GDD ≥4.5:1; feeds P6.3 audit.
- **Privacy** — Local preference only.
- **Maintainability** — Views reference semantic colors (`AppColors.cellBorder`) not raw hex; contrast variants live in P1.4 tokens.
- **Reliability** — Default (off) path unchanged from current MVP appearance.

## 7. Acceptance Criteria

- **AC-1.** *Given* high contrast enabled, *when* the board renders, *then* cell/region borders are visibly heavier than default mode.
- **AC-2.** *Given* high contrast enabled, *when* contrast is measured on primary text/icons, *then* ratio is ≥ 4.5:1.
- **AC-3.** *Given* high contrast toggled in Settings, *when* returning to the game, *then* the board reflects the new mode immediately.
- **AC-4.** *Given* high contrast enabled, *when* regions are viewed in grayscale simulation, *then* patterns still distinguish regions (P3.10).
- **AC-5.** *Given* a rule violation, *when* highlighted in high contrast, *then* violation state is clearly distinguishable from normal cells.

## 8. Data Model

No new model. Reads `SettingsStore.highContrastEnabled` (P4.3). Uses high-contrast variants from P1.4 tokens.

```swift
// UI/DesignSystem/Colors.swift
enum AppColors {
    static func cellBorder(highContrast: Bool) -> Color
    static func primaryText(highContrast: Bool) -> Color
    // ...
}
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Some pairs fail 4.5:1 | M | H | Audit all token pairs with contrast tool before merge |
| Patterns lost in high contrast | M | M | Test grayscale + high contrast together |
| Per-view hardcoded colors bypass tokens | M | M | Grep for raw `Color(` in views; enforce token usage |

## 15. Rollout Plan

- No feature flag. Toggle in Settings; default off.
- Rollback: disable token branching; default appearance returns.

- **Accessibility** — Grayscale region test + violation visibility (feeds P6.3).

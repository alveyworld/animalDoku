## 1. Problem Statement

In Mark mode, players mark impossible cells one tap at a time. During deduction this is tedious: clearing a row or region often means many identical mark toggles. Supporting click-and-drag (press and slide across cells) to paint marks in one gesture matches how similar logic puzzles feel and keeps the calm flow without slowing the player down.

## 2. Goals

- Let the player **press and drag** across the board in Mark mode to apply marks to every cell entered.
- Keep single-tap Mark behavior unchanged (toggle still works).
- Make the drag gesture feel precise: one logical action per cell entered, no double-toggles while lingering.
- Preserve Place mode’s existing tap semantics (no accidental places while dragging unless explicitly in scope — see Non-Goals / Open Questions).

## 3. Non-Goals

- Drag-to-place animals in Place mode (optional follow-up; default out of scope).
- Multi-touch / two-finger gestures.
- Undo collapsing an entire drag into a single undo step (nice-to-have; may ship as one undo entry — see Open Questions).
- Changing Mark toggle rules for animal-occupied cells (still no effect per Formal Rules).

## 4. Personas & User Stories

- **As a player**, I want to drag across empty cells in Mark mode so that I can X out a whole line quickly.
- **As a player**, I want a single tap to still toggle one cell so that precise edits stay easy.
- **As a player**, I want dragging not to flip the same cell repeatedly when my finger jitters in place.

## 5. Functional Requirements

- **FR-1.** In **Mark** mode, a press-and-drag gesture MUST apply the mark action to each **new** cell the pointer enters.
- **FR-2.** The drag stroke’s paint polarity MUST be determined by the **first cell** of the gesture:
  - If the first cell becomes (or is intended to become) **blocked**, subsequent empty cells in the stroke become blocked.
  - If the first cell becomes (or is intended to become) **empty** (clearing an X), subsequent blocked cells in the stroke clear to empty.
  - Cells where Mark has no effect (e.g. animal) MUST be skipped without ending the stroke.
- **FR-3.** Single-tap Mark behavior MUST remain: tap empty → blocked; tap blocked → empty; tap animal → no effect.
- **FR-4.** While dragging, entering the same cell again MUST NOT re-toggle it (visit once per stroke).
- **FR-5.** Drag MUST NOT fire when the board is locked (completed puzzle).
- **FR-6.** Place mode MUST ignore mark-drag painting (taps only), unless a follow-up story adds drag-to-place.
- **FR-7.** Haptics (P5.6), if enabled, SHOULD fire at most lightly per newly painted cell or once per stroke start — not a continuous buzz.

## 6. Non-Functional Requirements

- **Performance** — Drag tracking at 60fps; hit-testing cells under the finger without dropping frames on 8×8.
- **Accessibility** — VoiceOver users keep tap-to-toggle; drag is a pointer enhancement, not the only path. Announce or don’t trap focus.
- **Privacy** — N/A.
- **Maintainability** — Gesture lives on `BoardView` (or a thin overlay); cell mutation still goes through `GameViewModel` / `GameSession` Mark rules.
- **Reliability** — Finger lift ends the stroke cleanly; cancelled gestures leave a consistent board.

## 7. Acceptance Criteria

- **AC-1.** *Given* Mark mode and an empty board region, *when* the player presses on an empty cell and drags across several empty cells, *then* all entered empty cells become blocked.
- **AC-2.** *Given* Mark mode and a line of blocked cells, *when* the player presses on a blocked cell and drags across other blocked cells, *then* those cells clear to empty.
- **AC-3.** *Given* a drag stroke, *when* the finger remains inside one cell, *then* that cell is only modified once for the stroke.
- **AC-4.** *Given* Place mode, *when* the player drags across cells, *then* no marks are painted (and animals are not placed by drag).
- **AC-5.** *Given* a completed / locked board, *when* the player attempts to drag, *then* no cell states change.
- **AC-6.** *Given* Undo after a multi-cell drag, *when* Undo is tapped, *then* either the entire stroke reverts as one step **or** each cell undoes in reverse (product choice documented in Open Questions) — must be deterministic and tested.

## 8. Data Model

No new persisted model. Session/ephemeral stroke state only:

```swift
// Engine or ViewModel — ephemeral
struct MarkDragStroke {
    let paintBlocked: Bool          // polarity from first actionable cell
    var visited: Set<Position>
}
```

## 10. UI / UX

- Gesture: `DragGesture(minimumDistance: …)` on the board grid (coordinate → cell), or simultaneous with existing taps without breaking tap.
- Visual: existing blocked X appears as cells update (reuse P5.1 transitions if cheap).
- Modes: Mark only for paint; Place unchanged.
- States: idle → stroke active → stroke ended.
- Conflict: ensure tap still registers when the finger doesn’t move past the drag threshold.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Drag steals taps | M | H | Use minimumDistance; keep Button/tap path for zero-move |
| Jitter double-toggles cells | M | M | `visited` set per stroke (FR-4) |
| Hit-testing wrong cell under finger | M | M | Convert global location → row/col via board geometry |
| Undo spam after long stroke | M | M | Decide batch-vs-per-cell undo early; test AC-6 |
| Place mode accidental marks | L | M | Gate paint on `inputMode == .mark` |

## 15. Rollout Plan

- Ships in v1.1 polish; no feature flag required (behavior additive in Mark mode).
- Rollback: remove drag handlers; taps remain.

## 16. Test Plan

- **Unit** — Stroke polarity from first cell; visited set prevents re-entry; Place mode / locked board no-ops.
- **UI / integration** — Simulate drag path across positions; assert final cell states.
- **Manual** — Feel test on device: mark a row, clear a row, mix animal cells in path, tap vs. short drag.
- **Accessibility** — Confirm VoiceOver tap-to-mark still works without requiring drag.

## 17. Documentation & Training

- Note Mark drag polarity (paint vs. erase from first cell) in Formal Rules / player-facing help if tutorial mentions Mark (P5.5 may add a short line later).

## 18. Open Questions

1. **Undo:** One undo step for the whole stroke vs. one step per cell? **Decided: one undo step per stroke** (`GameAction.markStroke`).
2. **Place mode drag:** Defer drag-to-place animals to a later story? **Yes — deferred.**
3. **Erase-only vs. paint-only modifiers:** Any need for forced paint/erase without polarity-from-first-cell? **No — polarity-from-first is enough.**
4. **Haptics:** Light place haptic once when stroke polarity is set (first actionable cell), not per cell.

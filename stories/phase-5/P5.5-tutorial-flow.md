## 1. Problem Statement

The GDD goal is "easy to learn in under a minute," but tutorial content, length, and skip behavior are still open design decisions. New players may not understand the four rules or Place/Mark modes without guidance. A tutorial flow introduces core mechanics without blocking experienced players — but **design is TBD**, so this story defines the implementation shell and acceptance criteria pending product sign-off.

## 2. Goals

- Deliver a first-launch tutorial (or skippable onboarding) covering the four rules and Place/Mark modes.
- Let experienced players skip and never see it again (unless reset from settings).
- Keep total tutorial time under one minute per GDD goal.
- Persist completion/skip state locally.

## 3. Non-Goals

- Interactive puzzle-walkthrough with forced moves (unless design chooses it — see Open Questions).
- Video or animated rule explanations beyond simple static/step screens.
- Re-teaching on every launch.
- Hint system tutorial (hints are self-explanatory in toolbar).

## 4. Personas & User Stories

- **As a new player**, I want a quick explanation of the rules so that I can start playing confidently.
- **As a returning player**, I want to skip tutorial so that I go straight to the puzzle.
- **As a product owner**, I want a flexible tutorial container so that copy and steps can change without code rewrites.

## 5. Functional Requirements

- **FR-1.** On first launch (tutorial not completed/skipped), the app MUST present the tutorial before or over the first game session.
- **FR-2.** The tutorial MUST cover: one animal per row/column/region, no-touch rule, Place vs. Mark mode, and how to win.
- **FR-3.** The player MUST be able to skip the tutorial at any step.
- **FR-4.** Completing or skipping MUST persist `tutorialCompleted` (or equivalent) so it does not reappear.
- **FR-5.** A settings entry SHOULD allow replaying the tutorial (optional for v1.1).
- **FR-6.** Tutorial content MUST be driven from localizable strings/config so design can iterate without structural changes.

## 6. Non-Functional Requirements

- **Performance** — Tutorial screens are lightweight; no puzzle engine load required for static steps.
- **Accessibility** — All steps VoiceOver-readable; Dynamic Type; skip control ≥44pt.
- **Privacy** — Only a local boolean flag; no analytics required for MVP of tutorial.
- **Maintainability** — Steps defined as data (`TutorialStep` array) not hard-coded view hierarchy per sentence.
- **Internationalization** — All copy externalized.

## 7. Acceptance Criteria

- **AC-1.** *Given* first install, *when* the app launches, *then* the tutorial is presented.
- **AC-2.** *Given* the tutorial, *when* Skip is tapped, *then* the player reaches gameplay and `tutorialCompleted` is true.
- **AC-3.** *Given* a subsequent launch, *when* the app opens, *then* the tutorial does not appear.
- **AC-4.** *Given* the tutorial flow, *when* timed by QA, *then* a new user can complete it in under 60 seconds (design-dependent).
- **AC-5.** *Given* VoiceOver, *when* traversing tutorial steps, *then* all text and controls are announced.

## 16. Test Plan

- **Unit** — `tutorialCompleted` persistence; step array loads.
- **UI** — First launch shows tutorial; skip completes; second launch skips tutorial.
- **Manual** — Time-to-complete under 60s with final copy.
- **Accessibility** — VoiceOver on all steps.

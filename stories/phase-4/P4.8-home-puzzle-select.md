## 1. Problem Statement

The MVP launches directly into a single default puzzle with no home screen. As the catalog grows to 5–10 puzzles (v1.0) and beyond, players need a way to browse and choose a puzzle and filter by difficulty label. A `HomeView` provides puzzle selection and becomes the app's entry point in v1.1.

## 2. Goals

- Add a `HomeView` listing bundled puzzles with difficulty labels.
- Let the player start any bundled puzzle from the list.
- Make Home the app entry point for v1.1 (replacing direct-to-game).
- Surface a resume affordance for puzzles that have a saved attempt (via P4.7).

## 3. Non-Goals

- Puzzle generation (Future) — list is bundled puzzles only.
- Statistics, streaks, daily puzzles (Future).
- Board-size scaling by difficulty (difficulty is metadata/label only per GDD).
- Redesigning the Game screen (P3.14).

## 4. Personas & User Stories

- **As a player**, I want to pick which puzzle to play so that I'm not limited to one board.
- **As a player**, I want to filter by difficulty so that I can choose an appropriate challenge.
- **As a returning player**, I want to resume an in-progress puzzle from the list.

## 5. Functional Requirements

- **FR-1.** Home MUST list all bundled puzzles loaded via `PuzzleLoader`, each showing its difficulty label.
- **FR-2.** Selecting a puzzle MUST navigate to the Game screen initialized with that puzzle.
- **FR-3.** Home SHOULD provide a difficulty filter (Easy/Medium/Hard/Expert) that narrows the list.
- **FR-4.** Puzzles with a saved attempt (P4.7) SHOULD show a Resume affordance that restores state.
- **FR-5.** For v1.1, the app entry point MUST be Home; the Game screen is pushed/presented from it.
- **FR-6.** An empty catalog state MUST render gracefully (message rather than a blank list).

## 6. Non-Functional Requirements

- **Performance** — Puzzle metadata loads without blocking UI; scroll stays at 60fps.
- **Accessibility** — List rows and filters are VoiceOver-labeled, ≥44pt targets, Dynamic Type friendly.
- **Privacy** — No data collection.
- **Reliability** — A malformed puzzle file is skipped, not fatal, and logged.
- **Maintainability** — Home reads from `PuzzleLoader`; adding puzzles requires no Home code changes.
- **Internationalization** — Difficulty labels and titles localized.

## 7. Acceptance Criteria

- **AC-1.** *Given* the bundled catalog, *when* Home loads, *then* every valid puzzle appears with its difficulty label.
- **AC-2.** *Given* a puzzle row, *when* tapped, *then* the Game screen opens with that puzzle.
- **AC-3.** *Given* a difficulty filter selection, *when* applied, *then* only matching puzzles are listed.
- **AC-4.** *Given* a puzzle with a saved attempt, *when* Home shows it, *then* a Resume action restores the saved state.
- **AC-5.** *Given* an empty or all-invalid catalog, *when* Home loads, *then* an empty-state message is shown.

## 8. Data Model

No new persisted model. Reads `[Puzzle]` from `PuzzleLoader`; consults `SaveGameStore` (P4.7) for resume flags.

```swift
// UI/Screens/HomeView.swift
struct HomeView: View {
    let puzzles: [Puzzle]
    @State var difficultyFilter: Difficulty?  // nil = all
}
```

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| One bad puzzle file breaks the list | M | M | Skip invalid puzzles; show valid ones |
| Entry-point change regresses MVP flow | M | M | Gate behind v1.1; keep direct-to-game path for MVP builds |
| Resume state stale vs saved data | L | M | Read save freshly on appear |

## 15. Rollout Plan

- Ships in v1.1 behind a `homeScreenEnabled` flag; MVP/v1.0 keeps direct-to-game.
- Rollback: disable flag; launch straight into default puzzle.

## 16. Test Plan

- **Unit** — Filtering logic returns correct subsets; invalid puzzles excluded.
- **Integration** — Selecting a puzzle initializes Game with the right `Puzzle`; resume restores save.
- **UI** — Navigate Home → Game → back; empty-state renders.
- **Accessibility** — VoiceOver over rows/filters (feeds P6.3).

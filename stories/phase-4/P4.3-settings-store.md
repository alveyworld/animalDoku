## 1. Problem Statement

Player preferences — selected theme, sound on/off, high-contrast on/off — must persist across launches for a consistent experience. Without a single persistence layer, each feature would reach into `UserDefaults` independently, causing key collisions and inconsistent defaults. `SettingsStore` centralizes these preferences behind one observable type.

## 2. Goals

- Provide an observable `SettingsStore` persisting `selectedThemeId`, `soundEnabled`, and `highContrastEnabled`.
- Back persistence with `UserDefaults` / `@AppStorage`.
- Define sensible defaults (`frogs`, sound on, high contrast off).
- Expose the store app-wide so views bind directly to it.

## 3. Non-Goals

- `SaveGame` (in-progress puzzle) persistence — that is P4.7.
- The settings sheet UI (P4.4) or theme picker (P4.2).
- iCloud/cloud sync of preferences.

## 4. Personas & User Stories

- **As a player**, I want my theme, sound, and contrast choices remembered so that I don't reconfigure every launch.
- **As a developer**, I want one settings type so that persistence keys and defaults live in one place.
- **As a player using accessibility**, I want high-contrast preference honored on first paint after launch.

## 5. Functional Requirements

- **FR-1.** The store MUST persist `selectedThemeId: String` (default `"frogs"`).
- **FR-2.** The store MUST persist `soundEnabled: Bool` (default `true`).
- **FR-3.** The store MUST persist `highContrastEnabled: Bool` (default `false`).
- **FR-4.** Changes MUST be written immediately and survive app termination and relaunch.
- **FR-5.** The store MUST be observable so SwiftUI views update when a value changes.
- **FR-6.** On first launch with no stored values, defaults from FR-1–FR-3 MUST apply.

## 6. Non-Functional Requirements

- **Performance** — Reads/writes are synchronous key-value; negligible cost.
- **Security** — No sensitive data; standard `UserDefaults` acceptable.
- **Privacy** — Preferences stay on-device; no collection.
- **Reliability** — Corrupt/missing values fall back to defaults, never crash.
- **Maintainability** — Keys namespaced (e.g. `settings.selectedThemeId`); single file `Services/SettingsStore.swift`.

## 7. Acceptance Criteria

- **AC-1.** *Given* a fresh install, *when* the store is read, *then* `frogs`, sound on, contrast off are returned.
- **AC-2.** *Given* `soundEnabled` set to `false`, *when* the app relaunches, *then* the store reads `false`.
- **AC-3.** *Given* a bound SwiftUI view, *when* `highContrastEnabled` changes, *then* the view re-renders.
- **AC-4.** *Given* an unexpected/absent stored value, *when* read, *then* the default applies without crashing.

## 8. Data Model

```swift
// Services/SettingsStore.swift
@Observable
final class SettingsStore {
    var selectedThemeId: String       // default "frogs"
    var soundEnabled: Bool            // default true
    var highContrastEnabled: Bool     // default false
}
```

Backed by `UserDefaults` keys: `settings.selectedThemeId`, `settings.soundEnabled`, `settings.highContrastEnabled`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Key collision with other defaults | L | M | Namespace all keys under `settings.` |
| Default drift between store and UI | L | M | Single source of defaults in store |
| Migration issues if keys renamed later | L | M | Keep keys stable; document any renames |

## 15. Rollout Plan

- No feature flag. Merge before P4.2/P4.4.
- Rollback: revert store; features use inline defaults temporarily.

## 16. Test Plan

- **Unit** — Set each value, re-instantiate store against same defaults suite, assert persistence; assert defaults on empty suite.
- **Integration** — Bound view updates on change.
- **Manual** — Toggle values, kill and relaunch app, verify retained.

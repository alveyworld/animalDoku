## 1. Problem Statement

The game state stores `elapsedSeconds`, but nothing increments it, pauses it on backgrounding, or displays it. The Formal Rules require the timer to be tracked from v0.1 and shown on the win screen in v1.0. A `TimerService` owns elapsed-time tracking with correct background/foreground behavior, and the win screen renders the final time.

## 2. Goals

- Track `elapsedSeconds` for the active puzzle attempt, starting at first interaction (or session start).
- Pause the timer when the app enters background; resume on foreground.
- Stop the timer on puzzle completion so the win time is stable.
- Display the elapsed time on the win screen (v1.0).

## 3. Non-Goals

- Showing a live timer on the Game screen during play (GDD keeps it hidden until win).
- Time-based scoring, leaderboards, or par times.
- Persisting elapsed time across app kills — that is part of `SaveGame` (P4.7).

## 4. Personas & User Stories

- **As a player**, I want to see how long the puzzle took so that I have a light sense of accomplishment.
- **As a player who backgrounds the app**, I don't want time spent away to count against me.
- **As a developer**, I want a testable timer abstraction independent of wall-clock flakiness.

## 5. Functional Requirements

- **FR-1.** The service MUST accumulate `elapsedSeconds` while the puzzle is active and unsolved.
- **FR-2.** The service MUST pause on app background (`scenePhase` inactive/background) and resume on foreground.
- **FR-3.** The service MUST stop accumulating once the puzzle is completed.
- **FR-4.** Reset (per Formal Rules) MUST zero the timer along with board/history/hints.
- **FR-5.** The win screen MUST display the final elapsed time formatted as mm:ss (v1.0).
- **FR-6.** `elapsedSeconds` MUST remain the source value stored in `GameSession`.

## 6. Non-Functional Requirements

- **Performance** — Timer uses a lightweight tick; no busy-loop, no measurable battery impact.
- **Reliability** — Background/foreground transitions never double-count or skip; monotonic accumulation.
- **Privacy** — No data collection.
- **Accessibility** — Win-screen time is VoiceOver-readable (e.g. "2 minutes 5 seconds") and scales with Dynamic Type.
- **Maintainability** — Time source injectable for deterministic tests (no direct `Date()` in logic paths).
- **Internationalization** — Time format respects locale where applicable.

## 7. Acceptance Criteria

- **AC-1.** *Given* an active puzzle, *when* one second of active play elapses, *then* `elapsedSeconds` increments by 1.
- **AC-2.** *Given* the app enters background for 10s, *when* it returns, *then* `elapsedSeconds` did not advance during background.
- **AC-3.** *Given* the puzzle is completed, *when* time passes, *then* `elapsedSeconds` no longer increments.
- **AC-4.** *Given* reset, *when* invoked, *then* `elapsedSeconds` is 0.
- **AC-5.** *Given* completion at 125s, *when* the win screen shows, *then* it displays "02:05".

## 8. Data Model

```swift
// Engine/TimerService.swift
final class TimerService {
    private(set) var elapsedSeconds: Int
    func start(); func pause(); func resume(); func stop(); func reset()
}
```

`elapsedSeconds` mirrors the field in `GameSession` (Formal Rules — Game State).

- Must ship after: P3.1 (`GameSession` holds `elapsedSeconds`)

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Double counting across background cycles | M | M | Pause/resume via single scenePhase observer; test transitions |
| Timer drift vs wall clock | L | L | Accumulate from monotonic source; acceptable ±1s |
| Timer keeps running after win | M | M | Stop on completion flag |

## 15. Rollout Plan

- No feature flag. Tracked silently; surfaced only on win screen in v1.0.
- Rollback: hide time on win screen; keep tracking.

## 16. Test Plan

- **Unit** — With injected clock: start/tick/pause/resume/stop/reset transitions produce correct `elapsedSeconds`.
- **Integration** — Completion stops timer; reset zeroes it.
- **Manual** — Background app mid-play; confirm no counting; verify mm:ss on win.
